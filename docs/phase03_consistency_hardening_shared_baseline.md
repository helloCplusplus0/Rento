# Phase03 Consistency Hardening Shared Baseline

## 一、文档目的

本文档用于冻结 `phase03-consistency-hardening-*` 全部子任务共享的边界、允许路线、禁止路线与统一语义，避免后续 `/spec` 各自扩写出不同的解释。

## 二、共享前提

- `phase01-restart-foundation-*` 已完成
- `phase02-auth-gate-*` 已完成
- 当前默认工作流为 `phase03-consistency-hardening-*`
- 当前 UI 仍为冻结资产
- 当前数据库主线固定为 PostgreSQL
- 当前开发服务器由用户手动启动，AI 不主动后台运行 `npm run dev`

## 三、共享判断标准

- 默认优先贴合真实租务流程，不为了最简实现把业务事实压扁成理想化 CRUD 关系
- 默认优先事实表达，而不是优先追求最简表关系、最少状态字段或最省事的删除路径
- 默认要求业务真实，页面状态、接口状态和数据库事实应能互相解释
- 默认要求状态可解释，房间、合同、账单、仪表及其联动状态都应能映射回真实经营动作
- 默认要求历史可追溯，不通过级联删除或覆盖式更新抹平既有事实链
- 默认要求实现低复杂度，在满足以上原则前提下保持轻量、高效、低耦合

## 四、允许路线

- 允许优先通过服务端门禁拦截高风险删除和状态流转
- 允许把“删除”重定向为停用、终止、归档、解绑等更可追溯的动作
- 允许在不削弱最小门禁的前提下，为开发态收口统一的手动启动入口
- 允许修复与 `schema.prisma` 不一致的字段引用和查询条件
- 允许收口 `pendingAmount`、`receivedAmount` 与 `BillStatus` 的统一语义
- 允许把 SQLite 历史兼容写成显式兼容项，并补退出条件
- 允许更新与本阶段直接相关的顶层文档和阶段文档

## 五、禁止路线

- 禁止为了清理当前关系而批量物理删除 `Bill`、`BillDetail`、`MeterReading` 等历史事实
- 禁止把前端按钮禁用当作删除门禁的主要保证
- 禁止为了开发方便绕过 `middleware`、取消最小门禁或引入第二套认证路径
- 禁止把 `phase04` 的性能优化、辅助页分类、日志降噪混入本阶段主线
- 禁止在没有专项方案和验证的前提下直接重建迁移基线或切断兼容分支
- 禁止未经授权改动现有 UI 视觉风格

## 六、统一业务语义

### 6.1 删除与历史保留

- 删除房间、合同、账单、仪表时，优先判断其对合同链、账务链、抄表链的影响
- 若存在仍需保留的历史事实，默认动作应是拦截、停用、终止、归档或解绑
- 只有在确认不存在历史事实、追溯责任和业务门禁冲突时，才允许物理删除
- 若“最简关系”与“真实租务事实”冲突，默认以后者为准

### 6.2 账务字段

- `monthlyRent` 仅用于合同月租金语义
- `totalRent` 仅用于合同总租金语义
- `Room` 租金字段语义为 `rent`
- `pendingAmount` 表示未收金额
- `receivedAmount` 表示已收金额
- `BillStatus` 必须与上述金额字段的统计口径保持一致

### 6.3 多仪表历史

- `Meter` 是独立资产，不因解绑或停用而丢失历史
- `MeterReading` 与 `BillDetail` 的事实链优先保留
- 多仪表账单必须支持回溯到单个仪表明细

## 七、schema 与迁移基线

- `schema.prisma` 是当前数据主真相源
- `onDelete: Cascade` 是当前关系实现事实，不等于业务允许直接删
- `migration_lock.toml` 当前仍属于历史兼容项
- `migrate-and-seed.sh` 中的 `sqlite -> db push` 分支当前也属于兼容项
- `phase03` 只负责显式化兼容项与退出条件，不默认承担完整迁移基线重建

## 八、统一验证要求

- 每个子任务至少执行：
  - `npm run lint`
  - `npm run type-check`
- 若变更涉及账单、合同、仪表、抄表任一主链，必须补至少一条可执行验证路径
- 若变更涉及开发态运行入口，还必须证明浏览器验证与脚本验证对认证 / 数据库上下文的假设一致
- 验证应优先证明：
  - 服务端门禁已经收紧
  - 历史事实没有被默认清空
  - 字段与统计语义已与 schema 和业务规则一致
  - 判断标准仍然满足业务真实、状态可解释、历史可追溯、实现低复杂度

## 九、阶段停顿门禁

- 当前阶段文档一旦产出完成，必须停止并等待用户审核
- 未经用户明确指示，不得直接进入 `phase03-consistency-hardening-01-*` 的 `/spec`
- 后续任何子任务都不得越过本共享基线单独扩张范围
