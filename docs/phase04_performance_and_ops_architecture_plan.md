## 二、当前阶段结论

### 2.1 已完成前提

当前已完成并可直接复用的前提包括：

* `phase01-restart-foundation-*`

* `phase02-auth-gate-*`

* `phase03-consistency-hardening-*` 已完成边界冻结、删除门禁、开发态运行入口、账务语义与迁移兼容项显式化

* 页面与核心 API 已具备最小认证闭环

* PostgreSQL 主线、环境入口与阶段文档口径已基本收口

### 2.2 当前真实阻塞面

基于当前代码与目录实况，`phase04` 的阻塞面已收束为以下几类：

* 列表查询与统计查询仍存在数据库侧过滤、分页、聚合不够收口的问题

* 健康检查、日志与基础运行指标已经存在，但仍缺统一口径和阶段级治理边界

* `src/app` 下的 `performance-*`、`layout-demo`、`components`、`business-flow-validation` 等辅助页面仍未完成分类与门禁治理

* 若继续把性能页、验证页、演示页与正式业务入口并列暴露，会增加导航噪音与发布边界不确定性

* 若在没有阶段边界的前提下直接做性能优化，容易把缓存、监控、页面分类与新功能扩写混成一次高复杂度改动

### 2.3 为什么现在进入 phase04

当前适合进入 `phase04-performance-and-ops-*`，原因如下：

* `phase03` 的职责是把主链行为与语义收回到可解释、可验证的服务端真相源

* 在主链一致性显式化后，性能与运维治理才有稳定的统计口径和行为前提

* 当前最值得继续收口的，已经不是主链语义本身，而是“如何更快、更稳、更清晰地运行与观察系统”

### 2.4 为什么现在不进入更大规模治理

当前仍不适合直接进入更大范围的 `phase05` 或基础设施扩张，原因如下：

* 当前可观测性与性能问题仍可在单仓库、单应用、轻量脚本范围内治理

* 若过早引入复杂监控平台、消息队列、分布式缓存或多服务拆分，会偏离低复杂度原则

* `phase04` 的目标是把现有结构运行得更稳、更可解释，而不是提前搭建重型平台

### 2.5 为什么当前不做 UI 重构

* 当前 UI 已被明确冻结为既有资产

* `phase04` 对辅助页面与入口的治理重点在“分类、门禁、导航边界”，不在重新设计视觉语言

* 任何页面治理都应优先通过入口标识、路由限制、导航收口和文档说明完成，而不是改 UI 风格

## 三、工作流定位

### 3.1 新增 `phase04-performance-and-ops-*`

当前最合适的工作流是：

```text
phase04-performance-and-ops-*
```

它的职责不是新增业务功能，也不是把系统扩写成重型运维平台，而是：

* 收口关键列表接口和统计接口的数据库侧性能路径

* 明确健康检查、日志与基础性能指标的统一治理口径

* 完成开发辅助页、性能页、演示页的分类、门禁与入口边界治理

* 在不改变主链语义的前提下，提高系统的可运行性、可观察性与可维护性

### 3.2 核心原则

`phase04-performance-and-ops-*` 必须坚持：

* 真实业务主链优先稳定

* 数据库侧过滤、分页与聚合优先

* 运维能力优先轻量、可解释、低耦合

* dev-only 页面必须与正式业务入口显式分层

* 可观测性优先服务问题定位，而不是堆叠工具

* UI 结构只做最小必要调整

* 低复杂度优先

## 四、架构边界

### 4.1 本阶段允许做的事

* 修正关键列表与统计接口中的低效查询、全量拉取后内存过滤、无分页或重复聚合问题

* 为健康检查、错误日志、基础性能指标补阶段级统一口径

* 允许对辅助页面做“开发辅助 / 运维治理 / 正式业务入口”的分类与门禁

* 允许补最小文档、最小注释和最小脚本说明，使运行边界更清晰

* 允许对导航与入口做不改变视觉语言的最小收口

### 4.2 本阶段暂不做的事

* 不回到 `phase03` 重新扩写主链业务语义

* 不直接引入新的监控平台、复杂 tracing 基础设施或外部 APM

* 不把缓存体系、性能压测平台、CI/CD 重构混入本阶段

* 不做大规模 UI 重构

* 不把开发辅助页面直接升级为正式产品功能

* 不为了调试方便绕过现有认证与门禁

### 4.3 服务端与运行边界优先级

当前需要优先冻结的边界如下：

* 列表与统计查询优先数据库侧过滤、分页与聚合，不再默认依赖全量查询后内存加工

* 健康检查、错误日志、基础性能指标应服务于问题定位，而不是各自维护第二套口径

* 开发辅助页、演示页、验证页必须先明确是否仅限开发环境或受门禁保护

* 正式业务入口不应继续被性能测试页、布局演示页、组件展示页污染

### 4.4 路由与目录边界

* `src/app/performance-test`、`performance-benchmark`、`performance-analysis` 当前应优先视为开发辅助候选入口

* `src/app/layout-demo`、`components`、`business-flow-validation` 当前应优先视为开发/验证辅助候选入口

* `src/app/system-health`、`data-consistency` 更接近运维治理页，但仍需明确其门禁和导航归属

* `phase04` 的重点是先分类和门禁化，而不是立刻决定所有辅助页的最终去留

推荐子任务顺序固定为：

```text
phase04-performance-and-ops-01-baseline-and-scope-freeze
phase04-performance-and-ops-02-query-performance-closure
phase04-performance-and-ops-03-observability-and-health-hardening
phase04-performance-and-ops-04-dev-only-entry-classification-and-gating
```

## 六、结论

当前最合理、最稳妥的下一步，不是直接进入零散优化，而是：

```text
先冻结 phase04 的阶段边界与共享基线
    ->
再按子任务顺序逐个进入 /spec
```

这能确保：

* 后续性能优化不会重新污染主链语义边界

* 后续运维能力不会演化为高复杂度、低回报的堆叠

* 后续辅助页治理不会误伤正式业务入口与既有 UI 风格

* 后续 `phase05` 若需要更大规模运维/性能升级，可直接继承本阶段结论
