## 二、当前阶段结论

### 2.1 已完成前提

当前已经具备、并可直接复用的前提包括：

* `phase01-restart-foundation-*` 已完成，顶层文档和目录治理已收口

* `phase02-auth-gate-*` 已完成，页面与核心 API 已具备最小认证闭环

* `phase03-consistency-hardening-*` 已完成，主链语义与删除门禁已稳定

* `phase04-performance-and-ops-*` 已完成，关键查询、基础观测与 dev-only 入口治理已完成当前阶段收口

* 当前仓库已存在基础移动端适配资产：
  * 响应式布局
  * 移动端底部导航
  * safe-area 与移动端表单优化
  * `manifest`
  * `appleWebApp` metadata

* `fix_008` 已完成移动端路线分析，结论为：若重新进入下一阶段，最优路径应是 **单 UI Web 主线下的受控安卓优先 PWA 交付闭环**

### 2.2 当前真实阻塞面

基于当前代码、`fix_008` 分析结论与历史失败经验，`phase05` 的真实阻塞面已收束为以下几类：

* 当前只有 `manifest` 和基础 PWA metadata，没有真正可交付的 PWA 闭环

* 缺少正式支持矩阵，无法明确回答：
  * 支持哪些手机
  * 支持哪些浏览器
  * 是否承诺 iOS
  * 不支持时如何退化

* 缺少 service worker 策略、更新策略与缓存边界，若直接实现极易重现旧版本缓存脏读、行为不稳定和开发环境调试受阻问题

* 当前移动端布局仍带有过渡性实现，如 `AppLayout` 依赖客户端判屏切布局，说明移动端 UI 仍未完全收口到统一表达

* 当前没有一套可解释的“安装 -> 启动 -> 登录 -> 更新 -> 离线退化”闭环说明，因此无法把手机端正式交付给真实用户长期使用

### 2.3 为什么现在进入 phase05

当前若要进入新的阶段工作流，适合进入 `phase05-pwa-delivery-*`，原因如下：

* `phase04` 已经完成“运行更稳、更快、更可观察”的基础治理，当前最适合继续收口的是“如何把现有单一 Web 主线以低复杂度方式真正落到手机上”

* `fix_008` 已经把问题性质从“是否做移动端”收口为“如何以最小复杂度完成可安装交付”

* 若继续长期停留在“响应式网页 + 基础 manifest”状态，移动端体验会持续处于半成品，且团队会不断在 PWA / Flutter / 原生 / 壳应用之间反复摇摆

* 当前项目定位是私有化经营后台，不是面向所有用户与所有设备分发的公网产品，因此更适合采用受控环境下的 PWA 路线，而不是上升到双端并行体系

### 2.4 为什么 phase05 不应等同于“做一个 App”

`phase05` 的职责不是把系统改造成原生移动应用，也不是进入“跨端产品化”大扩张，而是：

* 在已有 Next.js Web 主线之上补齐最小可安装闭环

* 把手机端正式定性为“可安装的私有管理 Web App”

* 明确支持矩阵、退化策略、更新策略和私有部署前提

* 在不引入第二套业务真相源和第二套 UI 主线的前提下，提升真实手机场景可用性

### 2.5 为什么现在不回退到 Flutter / 原生 / 重壳方案

当前仍不适合直接回退到 Flutter 或新增原生 Android / iOS 独立客户端，原因如下：

* 当前主线早已稳定为单仓库 Next.js Web，重回双端并行会直接破坏低复杂度原则

* 当前核心诉求不是重型端能力，而是：
  * 安装到安卓桌面
  * 像应用一样启动
  * 登录稳定
  * 页面可用
  * 不维护第二套系统

* 若在尚未证明“受控安卓最小 PWA 方案不可行”前直接跳到壳或原生，会过早引入第二条发布链路和新的维护负担

* 本阶段应该先证明：在当前受控场景下，PWA 是否能够稳定落地；若失败，再决定是否需要极薄 Android 壳作为下一阶段 Plan B

## 三、工作流定位

### 3.1 新增 `phase05-pwa-delivery-*`

当前最合适的下一阶段候选工作流是：

```text
phase05-pwa-delivery-*
```

它的职责不是新增业务功能，也不是搭建一个重型跨端平台，而是：

* 冻结 PWA 的正式支持矩阵、环境分层与退化策略

* 补齐安装、启动、图标、manifest、service worker、更新与最小离线壳闭环

* 收口关键移动端页面的可用性，使其符合真实现场使用需求

* 在不改变业务主链真相源的前提下，让手机端成为可解释、可安装、可维护的正式使用入口

### 3.2 核心原则

`phase05-pwa-delivery-*` 必须坚持：

* 单仓库、单 UI、单业务真相源优先

* 安卓优先、受控环境优先，而不是一开始追求全平台完美统一

* 渐进增强优先：核心业务必须先能作为普通 Web 正常使用，PWA 只作为增强层

* service worker 必须最小化、可解释、可回滚

* 更新策略必须清晰，不能让缓存逻辑凌驾于业务正确性之上

* 移动端优化优先解决真实经营场景可用性，而不是追求“像原生 App 一样无所不能”

* 低复杂度优先

## 四、架构边界

### 4.1 本阶段允许做的事

* 冻结受控环境下的正式支持矩阵：
  * 安卓优先
  * 支持浏览器优先级
  * 不支持环境的退化策略

* 实现最小可安装 PWA 闭环：
  * manifest 收口
  * 图标与启动体验校准
  * 安装提示或安装引导
  * service worker
  * 最小离线兜底页
  * 更新策略

* 针对关键业务页做移动端可用性审计与最小修复：
  * 列表页
  * 表单页
  * 详情页
  * 搜索页
  * 首页提醒与底部导航

* 允许对布局切换、safe-area、软键盘处理、视口行为做最小必要收口

* 允许更新与 PWA、安装与私有部署直接相关的顶层文档、环境说明与发布说明

### 4.2 本阶段暂不做的事

* 不新增 Flutter / 原生 Android / iOS 独立客户端

* 不重建一套专属移动端 UI 或第二套路由结构

* 不把“支持 PWA”扩张为“支持所有手机、所有浏览器、所有安装场景”

* 不引入完整离线数据库、复杂后台同步、推送中心、消息编排或系统级原生能力

* 不因为移动端交付而放松当前认证、安全边界与私有部署要求

* 不在没有支持矩阵和更新策略的前提下直接堆 service worker 缓存逻辑

### 4.3 浏览器与环境边界优先级

当前必须优先冻结的边界如下：

* 第一优先级：`Android + Chrome`

* 第二优先级：`Android + Edge` 与其他兼容 Chromium 的主流浏览器，但不承诺与 Chrome 完全同等级安装体验

* iOS 不作为本阶段正式承诺目标，只允许保留基础访问和必要安装说明，不作为核心交付验收条件

* 本地开发环境不默认依赖 service worker 验证日常页面开发；PWA 真正验收应落在受控测试环境、真机环境与最终私有部署环境

* 不支持 PWA 的浏览器必须自动退化为普通响应式 Web，而不是直接不可用

### 4.4 环境分层边界

* 本地开发环境：
  * 默认禁用 service worker 注册与缓存验证
  * 不把 PWA 行为作为日常页面开发前提

* 受控测试环境：
  * 用于真机验证安装、启动、更新、离线兜底与退化路径
  * 允许按专项测试启用 service worker

* 私有部署生产环境：
  * 必须满足 HTTPS 与受控网络前提
  * 沿用与受控测试环境一致的 PWA 交付与更新口径

### 4.5 退化策略与缓存边界

* 浏览器或设备不支持安装能力时，系统必须自动退化为普通响应式 Web，不阻断登录、查询、录入与其他在线主链操作

* service worker 不可用、注册失败、被手动清除或更新异常时，系统仍必须可回到普通 Web 访问路径

* service worker 允许缓存的边界仅限于：
  * 应用壳静态资源
  * `manifest`
  * 图标
  * 最小离线兜底资源

* service worker 不默认缓存：
  * 动态业务接口
  * 鉴权态业务页面响应
  * 其他可能成为业务真相源的动态数据

* 离线场景只允许提供最小离线兜底页，不把离线能力扩张为可写业务的本地真相源

### 4.6 路由与交付形态边界

* `src/app` 现有业务页面仍是唯一正式业务入口，不新增第二套路由树

* 安装态和浏览器态使用同一套业务主链与同一套 UI，只允许在导航、视口、软键盘、安装提示等边界上做最小差异处理

* `manifest`、service worker、离线页、安装提示等应服务于统一产品形态，而不是与正式业务页面形成第二套入口逻辑

* PWA 只是交付增强层，不能覆盖或替代正常浏览器访问主线

## 六、结论

当前若用户决定重新尝试 PWA，最合理、最稳妥的下一步，不是直接写 service worker，而是：

```text
先冻结 phase05 的支持矩阵、环境边界、退化策略与更新原则
    ->
再按顺序逐个进入安装壳、SW、移动端可用性与发布验收
```

这能确保：

* 不会再次把 PWA 变成“浏览器支持不确定 + 调试不可控 + 版本缓存脏读”的高风险试验

* 不会在移动端交付问题上重新引入第二套 UI 主线

* 不会让 service worker、安装逻辑和业务主链混成一套难以解释的行为

* 后续若发现受控安卓最小 PWA 仍不可行，也能清晰判断是否需要进入下一轮 `/plan` 评估更薄的 Android 壳方案
