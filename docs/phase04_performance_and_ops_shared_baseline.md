# Phase04 Performance And Ops Shared Baseline

## 一、文档目的

本文档用于冻结 `phase04-performance-and-ops-*` 全部子任务共享的边界、允许路线、禁止路线与统一判断标准，避免后续 `/spec` 各自扩写出不同解释。

## 二、共享前提

- `phase01-restart-foundation-*` 已完成
- `phase02-auth-gate-*` 已完成
- `phase03-consistency-hardening-*` 已完成
- 当前默认工作流已切换为 `phase04-performance-and-ops-*`
- 当前 UI 仍为冻结资产
- 当前数据库主线固定为 PostgreSQL
- 当前开发服务器由用户手动启动，AI 不主动后台运行 `npm run dev`

## 三、共享判断标准

- 默认优先保持主链业务真相源稳定，不为性能优化回退到模糊语义
- 默认优先数据库侧过滤、分页与聚合，而不是先做内存加工或前端兜底
- 默认优先问题定位能力，而不是堆叠复杂运维平台
- 默认要求运维能力轻量、可解释、低耦合
- 默认要求 dev-only 页面、运维治理页与正式业务入口显式分层
- 默认要求实现低复杂度，在满足以上原则前提下保持轻量、高效、灵活、低耦合

## 四、允许路线

- 允许优先优化关键查询路径中的数据库侧过滤、分页、排序与聚合
- 允许补最小健康检查、日志与基础性能指标说明
- 允许为 dev-only 页面补门禁、分类标识、导航收口与文档用途说明
- 允许更新与本阶段直接相关的顶层文档和阶段文档
- 允许补最小验证脚本或最小基准验证路径

## 五、禁止路线

- 禁止为了性能优化重新放松 `phase03` 已收紧的主链语义与门禁
- 禁止把缓存体系、消息队列、外部 APM、Tracing 平台等重型方案混入本阶段主线
- 禁止把开发辅助页继续当作正式业务入口扩写
- 禁止未经专项方案就改动数据库基线、迁移链或认证路径
- 禁止未经授权改动现有 UI 视觉风格

## 六、统一治理语义

### 6.1 查询性能

- 关键查询应优先数据库侧过滤、分页与聚合
- 若存在全量查询后内存过滤，应优先评估是否可迁移到数据库侧
- 性能优化应首先服务于关键接口和关键页面，而不是辅助页基准结果本身

### 6.2 运维与观测

- 健康检查、错误日志、基础性能指标至少要能支撑问题定位
- 当前阶段不追求“全量监控平台”，而追求“必要、稳定、可解释的最小观测能力”
- 文档、脚本与 API 对运行状态的描述必须一致

### 6.3 页面分类与门禁

- `performance-*`、`layout-demo`、`components`、`business-flow-validation` 默认视为 dev-only 或治理辅助候选入口
- `system-health`、`data-consistency` 更接近运维治理页，但仍应明确门禁与导航边界
- 是否保留页面，优先依据开发辅助价值与发布边界判断，而不是页面是否已经存在

## 七、统一验证要求

- 每个子任务至少执行：
  - `npm run lint`
  - `npm run type-check`
- 若变更涉及关键查询路径，必须补至少一条性能或查询行为验证路径
- 若变更涉及健康检查、日志或性能指标，必须证明问题定位口径更清晰
- 若变更涉及页面分类与门禁，必须证明正式业务入口不再被辅助页面污染
- 验证应优先证明：
  - `phase03` 已收口的主链语义没有被破坏
  - 关键查询路径更清晰或更高效
  - 运行治理能力更适合定位问题
  - dev-only 页面与正式入口边界更清楚

## 八、阶段停顿门禁

- 当前阶段文档一旦产出完成，必须停止并等待用户审核
- `phase04-performance-and-ops-01-*` 仅用于冻结共享边界、页面分类原则与验收口径，不得提前进入查询优化、观测补强或页面治理实现
- 未经用户明确指示，不得直接进入 `phase04-performance-and-ops-02-*` 及后续实现子任务
- 后续任何子任务都不得越过本共享基线单独扩张范围
