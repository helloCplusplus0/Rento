# Fix 012 Analysis - 移动端 UI 视觉分批收口方案

## 1. 问题摘要
- 对应问题：`fix_012`
- 问题级别：`P1`
- 是否阻断修复：否

本问题不是“某一个页面样式写错”，也不是“某一条业务链逻辑异常”，而是当前 Rento 在移动端仍存在一组分散的视觉体验问题：字体层级、卡片间距、表单节奏、弹窗宽高与滚动策略虽然已具备响应式基础，但整体更偏 PC 口径，移动端的可读性与紧凑度仍不稳定。  
与此同时，当前任务缺少 Figma 或像素级视觉参数，无法像传统 UI 交付那样一次性冻结最终效果，因此 `fix_012` 不适合按“单次分析 -> 单次实现 -> 一次验收”推进，而应按“边界冻结 -> 小批次 spec -> 实现 -> 截图对比验收”的方式分轮收口。

## 2. 根因结论
- 根因一：当前项目虽然已有移动端适配主线，但共享视觉基线仍不够收口。
  - 页面容器、安全区、底部导航、输入防缩放等能力已经存在，但字体层级、间距节奏、复杂弹窗尺寸策略、表单字段排列并未完全统一到少数共享层中，导致不同页面各自写 `sm:` / `md:` / `max-w-*` / `gap-*` 的情况较多。
- 根因二：当前移动端体验问题并不集中在单一页面，而是分散在“卡片 + 表单 + 弹窗 + 详情模板”四类高复用结构中。
  - 若不先把问题归类为共享模式，而是按单页零散修修补补推进，后续很容易重复劳动，并持续出现“这个页面改好了，另一个同类页面还是旧表现”的分叉。
- 根因三：当前任务缺少精确视觉原型，无法在分析阶段一次性冻结最终参数。
  - 这意味着 `fix_012` 的真正难点不在于“能不能立刻写样式”，而在于“如何建立一个可回看、可迭代、不会失控扩张的推进机制”。

## 3. 证据链
- 页面预展示链路：
  - 用户已在 `fix_012 issue` 中明确说明：PC 端当前基本符合预期，主要问题集中在移动端视觉体验，且优先关注“字体大小 -> UI 间距”，暂不做大幅结构破坏式改动。
  - `docs/rentoUI/` 已存在多张 Rento 当前移动端截图，且仓库内同时保留了若干商用公寓管理系统截图，可作为“真实现状 vs 参考体验”的对照素材。
- 当前共享布局与移动端基础能力：
  - 全局移动端安全区、输入防缩放、底部导航偏移等基础能力已集中在 `src/app/globals.css`。
  - 页面级统一容器与 sticky 页头已在 `src/components/layout/PageContainer.tsx` 落地。
  - 移动端底部导航、视口高度与键盘避让逻辑已分别在 `src/components/layout/UnifiedNavigation.tsx` 与 `src/components/layout/AppLayout.tsx` 落地。
- 当前组件实现现状：
  - 卡片基座 `src/components/ui/card.tsx` 已成为事实标准共享层，说明当前项目并非缺少 UI 基座，而是共享层之上的视觉口径仍有分散。
  - 复杂弹窗在不同业务组件中仍有较多个体化尺寸策略，例如 `src/components/business/ContractBillDueSummaryDialog.tsx` 已经演化出更适合移动端的宽度写法，但这套经验尚未系统扩散到其他弹窗。
  - 移动端表单共享层 `src/components/ui/mobile-form.tsx` 已存在，但当前并未成为所有核心表单的主路径；如合同、租客、续租等页面仍大量直接拼接 `Label + Input + Textarea + 原生 select`。
  - 账单列表页仍存在移动端和桌面端双实现分流，代表文件为 `src/components/pages/BillListPage.tsx`、`src/components/business/BillCardCompact.tsx` 与 `src/components/business/bill-card.tsx`，说明移动端视觉问题既有共享层不足，也有页面层单独分支带来的口径偏差。
- 关键代码位置：
  - 全局样式基线：`src/app/globals.css`
  - 布局容器：`src/components/layout/AppLayout.tsx`
  - 页面容器：`src/components/layout/PageContainer.tsx`
  - 详情页模板：`src/components/layout/DetailPageTemplate.tsx`
  - 导航与安全区：`src/components/layout/UnifiedNavigation.tsx`
  - 移动端表单共享层：`src/components/ui/mobile-form.tsx`
  - 共享弹窗层：`src/components/ui/dialog.tsx`、`src/components/ui/sheet.tsx`
  - 代表业务文件：
    - `src/components/pages/BillListPage.tsx`
    - `src/components/business/ContractForm.tsx`
    - `src/components/pages/RenewContractPage.tsx`
    - `src/components/pages/CheckoutContractPage.tsx`
    - `src/components/business/ContractBillDueSummaryDialog.tsx`
- 是否存在历史脏数据：不存在。
  - 本问题是视觉与交互体验收口，不涉及账务事实、合同事实、仪表历史或数据库字段修复。

## 4. 影响面分析
- 新创建合同：
  - 不影响合同数据正确性，但会影响移动端创建/编辑表单的阅读效率、输入效率与提交舒适度。
- 续租：
  - 不影响续租事实生成，但续租页属于复杂表单页，移动端的字体层级、区块间距、按钮排列体验会直接受影响。
- 手工生成账单：
  - 不影响账单事实，但会影响账单列表、账单详情、创建账单、弹窗确认等入口的移动端浏览与操作体验。
- 历史已生成账单：
  - 不影响历史数据本身，但会继续影响历史账单在移动端的可读性与截图观感。
- 统计 / 仪表盘：
  - 不改变统计口径，但仪表盘卡片、摘要区和快捷入口若移动端间距或字号层级失衡，会持续影响运营者快速扫读信息的效率。

## 5. 候选方案对比
### 方案 A
- 做法：
  - 不先冻结方法与范围，直接根据当前截图和体验反馈逐页改样式。
  - 哪个页面看起来不舒服就立即改哪个页面，直到主观上“差不多”为止。
- 优点：
  - 启动最快，短期可见改动最直接。
  - 对单一页面问题能快速止血。
- 风险：
  - 极易演变成无边界样式漂移，今天改一个页面，明天又要回头改另一个同类页面。
  - 很难沉淀共享规则，后续仍会重复劳动。
  - 在没有 Figma 的前提下，最容易把 fix 做成长期“边改边猜”的隐性 UI 重构。

### 方案 B
- 做法：
  - 先在 `analysis` 中冻结推进方法、优先级、共享层切口与验收机制。
  - 后续按“小批次 spec”推进，每一批只处理 1 到 3 个页面或一组同类组件。
  - 每一轮都以截图对比和人工反馈为验收依据，再进入下一批。
- 优点：
  - 最符合当前“无精确原型、需逐轮人工反馈”的任务现实。
  - 可以把移动端视觉优化控制在 fix 工作流内，避免一下子升级成全局重构。
  - 便于把成功经验沉淀到共享层，例如页面容器、表单字段、弹窗尺寸和卡片节奏。
- 风险：
  - 分批推进意味着不会一次性结束，需要用户和实现方持续协作。
  - 若批次切分不合理，仍可能出现“这批好了、下一批又换风格”的局部分叉。

### 方案 C
- 做法：
  - 直接把移动端 UI 视觉问题升级为新的 `phase06`，先做完整设计基线、全局组件规范和大规模统一改造。
- 优点：
  - 若确实要全站统一重订移动端视觉语言，这条路径最完整。
  - 适合未来形成更稳定的组件级设计系统。
- 风险：
  - 当前证据尚不足以证明项目已经需要新的系统级阶段。
  - 容易超出“字体大小 -> UI 间距 -> 小步收口”的现有约束，演化为大范围视觉重构。
  - 与当前 fix 闭环优先级不匹配，成本和心智负担都偏高。

## 6. 推荐方案
- 推荐原因：
  - 推荐采用 **方案 B：先冻结推进方法，再按小批次 spec 迭代收口**。
  - 这既承认当前没有 Figma、无法一次到位的现实，也能继续遵守项目当前的 fix 工作流，不让任务失控扩张。
  - `fix_012` 的第一版 `analysis` 不应伪装成“最终视觉方案”，而应明确它是“移动端视觉治理的执行规则文档”。
- 实施边界：
  - 保持 PC 和移动端仍然共用同一套页面与组件主线，不拆分两套 UI 模板。
  - 当前 fix 只针对移动端视觉可读性、间距节奏和局部布局舒适度，不重写业务结构和页面导航。
  - 优先优化顺序固定为：
    - 字体大小 / 字重 / 行高层级
    - 卡片与区块间距
    - 表单字段垂直节奏与按钮排列
    - 复杂弹窗的移动端宽高与滚动策略
  - 每一轮实现都应尽量优先落到共享层或同类组件，而不是只修孤立页面。
  - 推荐的首轮切口不是“全站所有页面”，而是按高频场景拆成若干小批次，例如：
    - 批次 1：账单列表 / 账单详情 / 账单相关弹窗
    - 批次 2：合同详情 / 合同表单 / 续租与退租页
    - 批次 3：仪表盘 / 房间列表 / 其他剩余高频页
- 明确不在本次修复范围内的内容：
  - 不在当前 fix 中建立一套新的完整设计系统。
  - 不在当前 fix 中重做页面信息架构或导航模式。
  - 不在当前 fix 中为移动端单独复制页面或引入第二套模板。
  - 不在当前 fix 中追求一次性冻结所有页面的最终像素级视觉参数。
  - 不把当前任务升级为新 `phase*`，除非后续证据明确表明已超出局部 fix 边界。

## 7. 数据修复策略
- 是否需要修历史数据：不需要。
- 若需要，修复范围：
  - 不适用。
- 若不需要，原因：
  - `fix_012` 不涉及数据库结构、历史业务事实或已生成数据的语义修正，影响范围仅限前端视觉与交互体验。

## 8. 验收标准
- `fix_012` 的第一阶段验收不是“全站移动端一次完美”，而是以下几点：
  - 已冻结一份可执行的 `analysis`，明确后续以“小批次 spec”推进，而不是无边界直接改全站。
  - 已明确首轮优先优化对象、共享层切口和不在范围内的事项。
  - 后续每一批次都必须满足：
    - 至少选定 1 到 3 个页面或一组同类组件
    - 有移动端前后对比截图或可视化验收依据
    - 字体层级和间距节奏相较当前版本有明确改善
    - PC 端未被明显误伤
    - 不引入新的横向溢出、按钮遮挡、输入难点按等移动端退化问题
- 若某一轮优化后，用户仍能明确指出“体验更差”或“风格漂移”，则该轮不能视为验收通过，必须继续收口或回退。

## 9. 回滚条件
- 若后续任一批次实现出现以下任一情况，必须停止继续扩张并回滚该批次：
  - 为了移动端视觉优化而引入第二套独立页面或组件模板
  - PC 端现有布局、视觉语言或操作路径被明显破坏
  - 移动端虽局部更美观，但出现新的内容溢出、滚动异常、按钮不可达或输入区被遮挡
  - 同类页面之间的字体、间距、按钮样式进一步分叉，导致整体风格更不一致
- 若推进过程中发现问题已不再是“局部视觉优化”，而是需要：
  - 全局组件规范重订
  - 全站页面结构重排
  - 新的移动端设计系统
  - 大量导航与信息架构重构
  则必须停止以 `fix_012` 继续推进，并回到 `/plan` 重新评估是否需要新的阶段工作流。
