# Fix 002 Analysis - 合同详情页关联账单一致性与状态统计问题

## 1. 问题摘要
- 对应问题：`fix_002`
- 问题级别：`P1 / fix-now`
- 是否阻断修复：是

本问题不是单个 UI 显示错误，而是两类共享业务语义同时出现漂移：
- 第一类：合同详情页、账单列表页等展示层没有复用统一账单状态语义，导致“已逾期”“已收款”等统计口径分叉。
- 第二类：退租页面“结算预览”与后端实际落库账单不是同一套计算来源，且前端可编辑的预览结果根本没有提交到后端，因此会出现“预览为 0、落库非 0”的必然偏差。

## 2. 根因结论
- 根因一：合同详情页的“历史账单”统计在组件本地手写判断，绕开了已有的统一账单状态语义工具，把“到期日已过且状态不等于 `PAID`”直接当作逾期，导致 `PENDING` 甚至 `COMPLETED` 都可能被误记为“已逾期”。
- 根因二：账单列表页、合同详情页、共享账单统计模块对 `PAID / COMPLETED / OVERDUE / PENDING` 的理解不一致，已有统一语义文件但未被展示层全面复用，形成“同一份账单在不同页面统计不同”的双重真相。
- 根因三：退租页面的结算预览和后端退租 API 使用两套不同的结算公式与字段口径；前端可编辑结算项只影响本地展示，不会进入请求体，因此后端总会按自己的公式重算并落库。
- 根因四：退租场景中“应收/应退/结清旧账单/生成退租结算账单”的职责边界未冻结：前端把未付账单纳入预览汇总，后端却把未付账单直接自动结清而不是并入退租结算账单，导致预览语义和最终落库语义天然不一致。

## 3. 证据链
- 页面预展示链路：
  - 合同详情页的账单统计直接在 `EnhancedContractDetail.tsx` 本地计算：
    - `paid = status === 'PAID'`
    - `pending = status === 'PENDING'`
    - `overdue = today > dueDate && status !== 'PAID'`
  - 这套逻辑并未复用 `src/lib/bill-semantics.ts` 中的 `OPEN_BILL_STATUSES / SETTLED_BILL_STATUSES / isSettledBillStatus()` 等共享口径。
  - 退租页面的“结算预览”在 `CheckoutContractPage.tsx` 中本地计算，并允许用户编辑每个应退/应收项：
    - 预览包含租金差额、押金退还、钥匙押金退还、未付账单汇总、损坏赔偿等项目
    - 用户修改后只更新 `editableItems` 和本地 `settlement.summary`
- 服务端实际创建链路：
  - 合同详情页数据来自 `contractQueries.findById()` 返回的原始 `bills`，没有独立共享统计投影层。
  - 账单列表页状态统计直接按原始 `bill.status` 计数，没有统一合并 `PAID / COMPLETED`。
  - 退租提交时，前端只发送 `checkoutDate / checkoutReason / damageAssessment / finalMeterReadings / remarks`，不包含用户编辑后的结算结果。
  - 退租 API 在 `src/app/api/contracts/[id]/checkout/route.ts` 中重新调用 `calculateCheckoutSettlement()`，按自己的公式计算 `refundAmount / additionalAmount`，并据此创建退租结算账单。
  - 退租 API 随后会把所有旧未付账单直接更新为 `PAID`，而不是把它们并入“退租结算账单”的最终金额。
- 数据模型映射：
  - 统一账单语义已存在于 `src/lib/bill-semantics.ts`：
    - 开放状态：`PENDING / OVERDUE`
    - 已结清状态：`PAID / COMPLETED`
  - 但合同详情页、账单列表页、共享统计模块并未统一复用这套语义。
  - 退租请求体也没有独立的“结算明细 / 最终确认金额”字段，当前只有“退租原因 / 日期 / 损坏费 / 最终抄表”等输入，导致前端编辑结果没有正式的数据通道。
- 关键代码位置：
  - 合同详情页统计：`src/components/business/EnhancedContractDetail.tsx`
  - 账单统一语义：`src/lib/bill-semantics.ts`
  - 账单列表页状态统计：`src/components/pages/BillListPage.tsx`
  - 共享账单统计：`src/lib/bill-stats.ts`
  - 退租前端预览与编辑：`src/components/pages/CheckoutContractPage.tsx`
  - 退租后端结算与落库：`src/app/api/contracts/[id]/checkout/route.ts`
- 是否存在历史脏数据：存在高概率。
  - 统计漂移问题会污染合同详情页、账单列表页、部分聚合视图的展示结果。
  - 退租结算问题会产生历史退租结算账单金额与用户确认预览不一致的事实记录。

## 4. 影响面分析
- 新创建合同：
  - 会直接影响合同详情页的历史账单统计展示。
  - 特别是“新建未付账单且 dueDate 为当天或早于当前时刻”时，可能被错误展示为“已逾期”。
- 续租：
  - 续租后的合同详情同样复用 `EnhancedContractDetail`，因此会继承相同的统计口径漂移。
- 手工生成账单：
  - 只要账单进入合同详情页或账单列表页，都会受当前状态统计口径影响。
- 历史已生成账单：
  - 历史合同详情页和账单列表的“已逾期 / 已收款 / 已结清”展示都可能存在偏差。
  - 已发生的退租操作，如果当时前端预览和后端落库不一致，可能已留下错误的 `OTHER` 退租结算账单金额。
- 统计 / 仪表盘：
  - 合同详情页、账单列表页、共享账单统计模块对逾期与已收款的口径不一致，后续任何依赖这些页面结果做业务判断的操作都可能被误导。
  - 若未来继续叠加数据统计分析功能，会放大这类双重真相风险。

## 5. 候选方案对比
### 方案 A
- 做法：
  - 只修合同详情页的“已逾期”判断，把 `today > dueDate && status !== 'PAID'` 改成 `status === 'OVERDUE'` 或 `OPEN_BILL_STATUSES + dueDate` 的局部修正。
  - 只在退租页面禁用前端编辑，避免用户误以为自己改了最终落库结果。
- 优点：
  - 改动量最小，能快速消除最明显的两个现象。
  - 风险集中在 UI 层，短期可控。
- 风险：
  - 账单列表页、共享统计模块、退租后端结算仍保留原有分叉口径。
  - 只是掩盖问题，没有建立统一的统计与结算真相源。
  - 后续仍可能在别的页面继续出现“看起来修好了，但其他入口还是错的”。

### 方案 B
- 做法：
  - 抽离统一的“账单状态展示统计口径”工具，明确：
    - 什么叫“已逾期”
    - 什么叫“已结清”
    - 什么叫“开放账单”
  - 合同详情页、账单列表页、共享账单统计模块统一复用同一套统计工具。
  - 抽离统一的“退租结算真相源”：
    - 前端预览与后端落库共用同一套结算计算器
    - 若允许编辑预览，编辑后的结果必须以正式字段提交并由后端验证后落库
    - 若不允许编辑影响最终落库，则前端必须退化为只读预览，不能给用户虚假的可编辑感
  - 明确退租场景中“旧未付账单结清”和“新退租结算账单”之间的职责边界，并保持前后端一致。
- 优点：
  - 能同时修复当前两类问题，并收口“账单状态统计”和“退租结算”两条共享语义主线。
  - 更符合当前项目“主链真相源单一、先分析后修复”的要求。
  - 后续若做数据统计分析，也能直接继承这次冻结的语义结果。
- 风险：
  - 修改面较大，至少会触及合同详情页、账单列表页、共享统计模块、退租页和退租 API。
  - 需要先明确“退租预览是否允许编辑最终结算结果”的产品边界，否则实现容易继续摇摆。

## 6. 推荐方案
- 推荐原因：
  - 推荐采用 **方案 B 的“统一统计口径 + 统一退租结算真相源”**。
  - 当前问题已经不是单页 bug，而是共享语义分叉：
    - 账单状态统计语义分叉
    - 退租结算预览与落库语义分叉
  - 若只做局部补丁，短期能消除表面现象，但无法关闭未来同类问题。
- 实施边界：
  - 在当前 fix 内，优先收口以下两条共享能力：
    - `Bill status presentation semantics`
    - `Checkout settlement calculator`
  - 合同详情页、账单列表页、共享账单统计模块改为统一复用共享统计工具。
  - 退租页面与退租 API 改为统一复用同一套结算计算器。
  - 产品边界已明确冻结为：
    - 退租结算预览中的应退、应收项目允许人工调整，这些调整项必须作为正式结算载荷提交给后端，并由后端在安全边界内验证后落库
    - 除已完全结清账单外，其他账单都允许“有限制地编辑本次退租结算处理金额”
    - 用户点击“确认退租并结算所有账单”后，退租结算账单与关联旧未付账单都应直接进入闭环态，不再保留为 `PENDING / OVERDUE`
    - 即使最终净额为 `0`，系统仍必须生成一张金额为 `0` 的退租结算账单作为审计留痕
    - 已完全结清账单固定为不可编辑：`PAID`、`COMPLETED`
    - 可受限编辑账单固定为未完全结清账单：`PENDING`、`OVERDUE`，以及部分收款但仍有待收余额的账单
  - “有限制编辑”的语义冻结为：
    - 编辑的是“本次退租结算决策”，不是直接改写原始账单历史事实
    - 单条应退项目本次纳入结算的退款金额不得小于 `0`
    - 单条应退项目本次纳入结算的退款金额不得大于该项目当前 `originalAmount`
    - 单条账单本次纳入结算的应收金额不得小于 `0`
    - 单条账单本次纳入结算的应收金额不得大于该账单当前 `pendingAmount`
    - 若账单已发生部分收款，不允许通过退租编辑抹除既有已收事实
    - 若旧未付账单未按原待收全额纳入本次退租结算，剩余差额应按“退租减免/豁免”写入审计线索并直接闭环
    - 人工调整必须保留调整原因、操作人和原始金额/调整金额的审计线索
  - 保持 Prisma schema 不做大改，优先在服务层与页面层收口共享语义。
- 明确不在本次修复范围内的内容：
  - 不在本次 fix 中重构整套账单模型或引入新状态机。
  - 不在本次 fix 中顺手开发完整的数据统计分析模块。
  - 不在本次 fix 中自动批量回写所有历史统计展示；历史错误退租结算账单需先识别、分层，再决定是否修复。

## 7. 数据修复策略
- 是否需要修历史数据：需要，但分两层处理。
- 若需要，修复范围：
  - 统计展示类问题：
    - 多数属于“展示口径漂移”，修复共享统计工具后可直接改善后续展示，不一定需要修改历史账单数据。
  - 退租结算账单一致性问题：
    - 需要识别历史 `OTHER` 类型、`paymentMethod='退租结算'` 或 `remarks` 含“退租结算”的账单
    - 对比当时合同、旧未付账单、损坏赔偿、最终抄表等事实，判断是否存在“预览逻辑和落库逻辑不一致”造成的异常金额
  - 仅建议对以下安全范围评估自动修复：
    - 退租结算账单仍为 `PENDING`
    - `receivedAmount = 0`
    - 未发生后续人工修改
    - 没有被其他流程引用
  - 对已支付、已完成、已人工修改或无法准确回溯原始预览意图的退租结算账单，只输出审计清单，不自动改写。
- 若不需要，原因：
  - 不适用。本问题已进入真实数据阶段，且退租结算账单会形成真实账务事实，不能只修未来路径而忽略历史审计。

## 8. 验收标准
- 合同详情页“历史账单 Tab”中的：
  - “已逾期”
  - “已收款 / 已结清”
  - “待处理”
  必须与统一账单状态语义一致，不再由页面各自手写条件。
- 账单列表页与共享账单统计模块对 `PAID / COMPLETED / OVERDUE / PENDING` 的展示与聚合结果一致。
- 新创建但未逾期的账单，不会在合同详情页被误显示为“已逾期”。
- 退租页面预览与后端实际落库账单必须来自同一套结算真相源。
- 用户调整后的最终结算结果必须真正参与后端校验与落库；不得继续出现“前端改成 0，后端仍按原公式落库”的情况。
- 应退项目允许在 `0 ~ originalAmount` 安全边界内调整本次退租结算退款金额，并进入正式审计线索。
- 退租确认后始终生成一张退租结算账单；当最终净额为 `0` 时，也必须生成金额为 `0` 的退租结算账单。
- 退租结算账单在创建后即进入 `COMPLETED` 闭环态，不再停留为 `PENDING`。
- 已完全结清账单在退租结算中不可编辑；未完全结清账单仅允许在 `0 ~ pendingAmount` 安全边界内调整本次结算金额。
- 若账单已发生部分收款，退租编辑后的结算结果不得破坏既有已收事实。
- 退租确认后，所有关联旧未付账单都必须被收口为闭环态；若本次纳入金额小于原待收，剩余差额按退租减免记录审计说明。
- 退租确认后必须保留结算调整的审计线索，至少包括：原始金额、调整金额、调整原因、操作人。
- 至少补一条主链验证路径：
  - 合同详情页存在一条新建未付账单
  - 预期“已逾期”统计为 0
- 至少补一条退租结算验证路径：
  - 退租预览与最终创建的退租结算账单金额完全一致
  - 旧未付账单处理方式与预览说明一致

## 9. 回滚条件
- 若统一统计工具接入后出现以下任一情况，必须立即停止发布并回滚：
  - 合同详情页与账单列表页统计结果更加分叉
  - `PAID / COMPLETED` 被错误合并或错误拆分，导致既有结清语义回归
  - 原本明确为 `OVERDUE` 的账单被错误展示为未逾期
- 若退租结算统一后出现以下任一情况，必须立即回滚：
  - 旧未付账单被重复结清或重复生成退租账单
  - 抄表结算账单、押金退还、损坏赔偿被误伤
  - 已存在的已支付/已完成退租结算账单被意外改写
- 若历史审计发现：
  - 无法准确回溯原始退租预览意图
  - 自动修复会破坏账务追溯
  - 影响面超出安全边界
  则必须降级为“只修未来路径 + 导出人工审计清单”，不得强行批量修复历史数据。
