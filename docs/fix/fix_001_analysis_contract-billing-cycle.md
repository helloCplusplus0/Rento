# Fix 001 Analysis - 合同支付周期与账单生成不一致

## 1. 问题摘要
- 对应问题：`fix_001`
- 问题级别：`P1 / fix-now`
- 是否阻断修复：是

本问题不是页面展示偏差，而是合同主链中的真实账单生成错误。  
在 `paymentMethod=半年付` 的真实场景下，创建合同页底部的账单预展示正确，但合同创建完成后的实际落库账单被错误拆成了 `12` 条月租账单，直接破坏了“支付周期 -> 账单生成策略”的主真相源。

## 2. 根因结论
- 根因一：前端预展示与后端实际生成使用了两套不同的支付周期解析与账期切分逻辑，没有单一真相源。
- 根因二：后端租金账单生成器在计算账期时，把合同里保存的原始中文 `paymentMethod` 直接传给只识别英文枚举值的 `calculateAllBillPeriods()`，导致 `半年付` 落入默认分支，被按月切分。
- 根因三：`Contract.paymentMethod` 当前既承载“合同支付周期”语义，又在其他路径被拿作“账单收款方式/特殊流程标签”使用，字段语义漂移提高了同类问题继续扩散的风险。

## 3. 证据链
- 页面预展示链路：
  - 合同创建表单的支付方式选项固定为中文值 `月付/季付/半年付/年付`，见 `src/components/business/ContractForm.tsx`。
  - 账单预展示在 `src/components/business/ContractBillPreview.tsx` 中自行解析中文支付周期：
    - `parsePaymentCycle('半年付') -> '半年付'`
    - `calculateRentAmount(monthlyRent, '半年付') -> monthlyRent * 6`
    - `calculateAllBillPeriods(startDate, endDate, '半年付') -> 6 个月一期`
  - 因此前端预展示正确，这与 `fix_001_issue_contract-billing-cycle.md` 中“界面底部预展示正确”的现象一致。
- 服务端实际创建链路：
  - 创建页通过 `/api/contracts` 提交合同时，原样保存 `paymentMethod` 到 `Contract.paymentMethod`，见 `src/app/api/contracts/route.ts`。
  - 合同创建完成后，异步调用 `generateBillsOnContractSigned(contractId)` 生成账单，见 `src/app/api/contracts/route.ts`。
  - 该函数在 `src/lib/auto-bill-generator.ts` 中先生成押金/钥匙押金/清洁费，再调用 `generateAllRentBills(contract)` 生成租金账单。
- 数据模型映射：
  - `Contract.paymentMethod` 在 Prisma schema 中是 `String?`，仅注释为“付款方式描述”，没有枚举或专门的支付周期字段，见 `prisma/schema.prisma`。
  - 当前同一业务语义至少存在三套值域：
    - 中文：`月付/季付/半年付/年付`
    - 小写英文：`monthly/quarterly/semi_annually/annually`
    - 大写英文：`MONTHLY/QUARTERLY/SEMI_YEARLY/YEARLY`
- 关键代码位置：
  - 后端租金账单生成函数 `generateAllRentBills(contract)`：
    - 先解析 `const paymentCycle = parsePaymentCycle(contract.paymentMethod || '月付')`
    - 但计算账期时却传入 `contract.paymentMethod || '月付'`
  - `calculateAllBillPeriods()` 的 `switch` 只识别 `QUARTERLY / SEMI_YEARLY / YEARLY / default`
  - 因此 `paymentMethod='半年付'` 时不会命中 `SEMI_YEARLY`，最终落到 `default`，按月拆账。
- 是否存在历史脏数据：存在高概率。
  - 所有通过以下路径复用 `generateBillsOnContractSigned()` 的流程都可能已产生同类错误：
    - 新建合同
    - 续租
    - 手工“重新生成账单”接口
  - 另外，`checkMissingRentBills()` 虽然传入了解析后的周期，但它与主生成逻辑并不共用同一账期真相源，后续仍有继续制造不一致的风险。

## 4. 影响面分析
- 新创建合同：
  - 直接受影响。
  - 非月付合同会在落库阶段被错误拆成月租账单，前端预展示与实际账单不一致。
- 续租：
  - 直接受影响。
  - `src/app/api/contracts/[id]/renew/route.ts` 在创建续租合同后复用同一个 `generateBillsOnContractSigned()`。
- 手工生成账单：
  - 直接受影响。
  - `src/app/api/contracts/[id]/generate-bills/route.ts` 直接调用同一个生成器。
- 历史已生成账单：
  - 可能已存在脏数据。
  - 特别是 `paymentMethod in ('季付','半年付','年付')` 且当前 RENT 账单条数与预期周期不匹配的合同，需要审计。
  - 已支付、部分支付、已完成的账单不能直接自动重建，否则可能破坏历史账务事实。
- 统计 / 仪表盘：
  - 当前账单总额理论上可能仍接近正确，但账单期数、到期日分布、逾期统计、待收账龄、提醒逻辑都会被错误账期污染。
  - 若后续依赖 RENT 账单条数或账期分布做统计，将继续放大偏差。

## 5. 候选方案对比
### 方案 A
- 做法：
  - 只在 `auto-bill-generator.ts` 中做局部补丁，把 `calculateAllBillPeriods()` 的入参从 `contract.paymentMethod` 改成 `parsePaymentCycle()` 的结果。
  - 顺手修掉 `calculateRentAmount()` 与备注文案的几个分支。
- 优点：
  - 修改量最小，能快速修复当前“半年付变月付”的直接问题。
  - 对外行为变化集中在一个文件，短期风险较低。
- 风险：
  - 前端预展示、后端生成、缺失账单补齐、其他账单计算工具仍然各自维护自己的周期值域和算法。
  - 下一个类似问题很可能以“季付/年付”、“续租/补账单”、“预展示/统计口径不一致”的形式再次出现。
  - 无法从根上消除“多套支付周期真相源”的结构性风险。

### 方案 B
- 做法：
  - 抽离统一的“合同支付周期标准化 + 账期切分 + 每期租金计算”工具模块，作为支付周期的单一真相源。
  - 约定内部标准口径，例如：
    - 内部标准：`MONTHLY | QUARTERLY | SEMI_YEARLY | YEARLY`
    - 输入兼容：中文值、历史英文小写值、历史英文大写值
    - 输出统一：标准周期值、周期月数、每期金额、账期数组
  - 合同创建预展示、后端自动生成、续租、手工重新生成、缺失账单补齐统一复用同一套逻辑。
  - 当前 fix 内不做数据库 schema 重构，只做兼容式语义收口；继续保留 `Contract.paymentMethod` 的存量值读取，但不允许各处再自行解释。
- 优点：
  - 能同时修复当前问题和相邻风险点，符合“合同主链单一真相源”的要求。
  - 可在不立即改 schema 的前提下最大限度降低回归风险。
  - 便于后续针对历史数据做一致性审计与修复。
- 风险：
  - 修改面比方案 A 更大，需要覆盖前端预展示、后端生成、续租和补账单路径。
  - 若实现边界控制不好，容易顺手扩成字段重构或大规模账务治理。

## 6. 推荐方案
- 推荐原因：
  - 推荐采用 **方案 B 的“兼容式单一真相源收口”**。
  - 本问题已经不是单点 bug，而是“合同支付周期”在主链上的真相源断裂。若只做局部补丁，短期能止血，但无法保证续租、补账单、统计或后续 fix 不再次踩到相同问题。
- 实施边界：
  - 在当前 fix 内，只收口“支付周期标准化 + 账期切分 + 每期租金计算”的共享逻辑。
  - 前端预展示与后端生成统一复用同一套周期工具。
  - 续租、手工重新生成账单、缺失租金账单补齐全部切到同一套逻辑。
  - 保持当前 Prisma schema 不变，不在本 fix 内直接新增/迁移 `paymentCycle` 字段。
  - 保持 UI 结构与表单交互基本不变，只修正语义与生成行为。
- 明确不在本次修复范围内的内容：
  - 不在本次 fix 中重构整套合同/账单数据模型。
  - 不在本次 fix 中顺手重做统计分析模块。
  - 不在本次 fix 中统一所有“账单收款方式”字段语义，只要求不再让它污染“合同支付周期”生成主链。
  - 不在本次 fix 中自动修改所有历史账单；历史脏数据处理必须分层、分条件执行。

## 7. 数据修复策略
- 是否需要修历史数据：需要，但必须分层处理。
- 若需要，修复范围：
  - 先做“识别与分级”，再决定自动修还是人工复核。
  - 建议把历史合同按以下条件分层：
    - `paymentMethod in ('季付','半年付','年付')`
    - RENT 账单条数与理论周期条数不一致
    - RENT 账单金额与理论每期金额不一致
  - 其中可自动修复的范围应严格限制为：
    - 所有错误 RENT 账单均为 `PENDING`
    - `receivedAmount = 0`
    - 未产生关联支付、逾期处理、人工备注依赖
  - 对已支付、部分支付、已完成、已人工修改的账单，只输出审计清单，不自动改写。
- 若不需要，原因：
  - 不适用。本问题已在真实数据中复现，且会直接污染合同账务事实。只修未来数据不足以关闭当前 fix。

## 8. 验收标准
- 合同创建页在以下支付周期下，预展示账单与实际落库账单完全一致：
  - `月付`
  - `季付`
  - `半年付`
  - `年付`
- 续租路径与手工 `/api/contracts/[id]/generate-bills` 路径在相同支付周期下生成结果一致，不再各自产生不同账期。
- `generateAllRentBills()`、`checkMissingRentBills()`、前端预展示不再各自维护不同的支付周期解释。
- 现有一次性费用账单（押金、钥匙押金、清洁费）不被误伤。
- 对已存在历史错误账单，至少能输出一份可执行的识别规则或修复清单；自动修复仅限“未支付且无收款历史”的安全范围。
- 至少补一条主链验证路径：
  - 新建 12 个月合同 + `半年付`
  - 预期生成 `2` 条 RENT 账单，每条金额 = `monthlyRent * 6`
- 至少再补一条相邻验证路径：
  - 新建 12 个月合同 + `季付` 或 `年付`
  - 验证生成条数和金额正确

## 9. 回滚条件
- 若统一周期工具接入后出现以下任一情况，必须立即停止发布并回滚：
  - `月付` 合同生成结果发生回归
  - 押金/钥匙押金/清洁费账单生成被误伤
  - 续租或手工生成账单路径出现重复账单或空账单
  - 非历史修复范围内的已支付账单被意外改写
- 若历史数据修复脚本在试运行中发现：
  - 影响范围超出“未支付且无收款历史”安全边界
  - 存在无法准确恢复原始周期的合同
  - 自动修复会破坏账务追溯
  则必须降级为“只修未来数据 + 导出人工审计清单”，不得强行批量改写历史记录。
