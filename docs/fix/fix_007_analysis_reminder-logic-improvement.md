# Fix 007 Analysis - 提醒模块逻辑梳理与优化

## 1. 问题摘要
- 对应问题：`fix_007`
- 问题级别：`P1 / design-review`
- 是否阻断修复：否

本问题不是单点接口错误，而是一次围绕 Dashboard 提醒模块的语义梳理与产品边界收口。当前首页提醒区同时摆放了 `空房查询`、`离店提醒`、`30天搬入`、`合同到期` 四类信息，但它们混合了三种不同性质：
- 第一类：客观状态类信号，例如空房、已到期未处理合同。
- 第二类：窗口型经营提醒，例如即将离店、即将生效。
- 第三类：历史遗留/未完全收口的配置与实现，例如“30天搬入”的硬编码窗口、提醒链路的双实现、旧设置键继续保留。

如果不先完成语义定性，就会继续出现以下问题：
- 用户难以判断四个按钮的区别，尤其是 `离店提醒` 与 `合同到期` 看似接近但实际并不等价。
- 设置页无法判断哪些提醒窗口应该进入全局配置，哪些属于客观状态、不应做成可调阈值。
- 源码中会继续同时存在“当前首页真实实现”和“旧版聚合实现/旧字段”，形成新的双重真相。

## 2. 根因结论
- 根因一：提醒模块缺少统一语义分层。当前四个提醒被并排展示，但实际上混合了 `房态信号`、`合同生命周期窗口提醒`、`已逾期待处理事项` 三种不同业务维度，导致命名和配置边界都不清晰。
- 根因二：窗口型提醒只完成了半套收口。`离店提醒` 已经统一读取 `contractExpiryAlertDays`，但 `30天搬入` 仍使用固定 `30` 天硬编码，形成“同属窗口提醒却只有一半可配置”的不一致。
- 根因三：`离店提醒` 与 `合同到期` 共用同一条合同结束日期主链，但前端文案没有明确表达“前者是即将到期窗口内，后者是已到期未处理”，用户容易误读为两个重复提醒。
- 根因四：提醒模块当前存在双实现痕迹。首页实际使用前端并发调用 4 个 Dashboard API 的实现，但服务端仍保留旧版聚合函数 `getDashboardAlerts()`，提醒逻辑存在重复装配路径。
- 根因五：提醒相关全局设置仍有历史遗留键继续停留在模型里，例如 `reminderDays`、`readingReminderDays`、`enableNotifications`，但它们并未进入当前 Dashboard 四个提醒的真实消费链，若继续暴露或继续保留模糊命名，会再次制造“设置存在但不生效”的问题。

## 3. 证据链
- 页面预展示链路：
  - 首页入口是 `src/app/page.tsx`，实际渲染的是 `src/components/pages/DashboardPageWithStats.tsx`。
  - 提醒面板实际组件是 `src/components/business/UnifiedAlertsPanel.tsx`。
  - 该组件固定渲染 4 个按钮：
    - `空房查询`
    - `X天离店`（标题来自 API 返回）
    - `30天搬入`
    - `合同到期`
  - 其中只有离店标题会随服务端设置动态变化；搬入标题和窗口都仍写死为 `30天`。
- 服务端实际创建链路：
  - `空房查询`：`src/app/api/dashboard/vacant-rooms/route.ts`
    - 条件是 `Room.status = VACANT`
  - `离店提醒`：`src/app/api/dashboard/leaving-tenants/route.ts`
    - 条件是 `Contract.status = ACTIVE`
    - `endDate` 落在 `today ~ contractExpiryAlertDays` 窗口内
    - 标题来自 `src/lib/contract-alert-semantics.ts`
  - `30天搬入`：`src/app/api/dashboard/upcoming-contracts/route.ts`
    - 条件是 `Contract.status = PENDING`
    - `startDate` 落在未来 30 天内
    - 当前没有读取全局设置
  - `合同到期`：`src/app/api/dashboard/contract-alerts/route.ts`
    - 条件是 `Contract.status = ACTIVE`
    - `endDate < today`
    - 本质是“已到期但未处理合同”，不是“即将到期”
- 数据模型映射：
  - 当前没有独立的 `Reminder` 领域模型。
  - 四个提醒全部建立在现有主链事实之上：
    - `Room.status`
    - `Contract.status`
    - `Contract.startDate`
    - `Contract.endDate`
  - 提醒配置真相源只有 `GlobalSetting` 中的 `contractExpiryAlertDays` 已真实接入。
  - `reminderDays`、`readingReminderDays`、`enableNotifications` 仍存在于设置模型或默认配置中，但未进入当前 Dashboard 四个提醒主链。
- 关键代码位置：
  - 首页提醒组件：`src/components/business/UnifiedAlertsPanel.tsx`
  - 离店提醒 API：`src/app/api/dashboard/leaving-tenants/route.ts`
  - 搬入提醒 API：`src/app/api/dashboard/upcoming-contracts/route.ts`
  - 合同到期 API：`src/app/api/dashboard/contract-alerts/route.ts`
  - 空房提醒 API：`src/app/api/dashboard/vacant-rooms/route.ts`
  - 合同提醒语义工具：`src/lib/contract-alert-semantics.ts`
  - 旧版提醒聚合实现：`src/lib/dashboard-queries.ts`
  - 设置页正式提醒配置：`src/app/settings/page.tsx`
  - 提醒相关全局设置真相源：`src/lib/global-settings.ts`
- 是否存在历史脏数据：
  - 不存在账单、合同、抄表那类“历史事实被写坏”的问题。
  - 当前问题是提醒语义、命名、配置入口和实现路径的设计收口问题，不需要做历史业务数据修复。

## 4. 影响面分析
- 新创建合同：
  - 会影响 Dashboard 的 `30天搬入` 与后续 `离店提醒 / 合同到期` 呈现逻辑。若提醒命名或窗口口径不清晰，新合同进入生命周期后会直接反映为首页认知噪音。
- 续租：
  - 会影响合同结束时间重算后的 `离店提醒` 和 `合同到期` 归类。若“即将到期”和“已到期未处理”继续混名，续租后的提醒判断将难以解释。
- 手工生成账单：
  - 基本不受影响。当前四个提醒不依赖账单生成主链，也不应为了对齐提醒模块而把 `billReminderDays` 强行拉入设置页。
- 历史已生成账单：
  - 不受影响。本问题不改变账务事实，也不涉及账单回补。
- 统计 / 仪表盘：
  - 影响显著。Dashboard 首页是该模块的唯一主要消费入口；若语义不收口，会持续影响日常运营判断与首页信息可信度。
- 设置页：
  - 影响显著。只有完成提醒分类后，才能明确哪些项属于“窗口型全局兜底配置”，哪些项不应进 Settings。
- 通知中心 / 未来提醒扩展：
  - 会影响后续规划。如果当前不先明确“提醒卡片”和“通知系统”是两种能力，后续很容易把 `enableNotifications`、`billReminderDays` 等未成型能力再次错误暴露出来。

## 5. 候选方案对比
### 方案 A
- 做法：
  - 保留当前四个提醒的布局与名称。
  - 仅补齐 `30天搬入` 的窗口配置化，让它像 `离店提醒` 一样从全局设置读取天数。
  - 其余逻辑不调整，旧聚合实现继续保留。
- 优点：
  - 改动最小，用户界面变化小。
  - 能快速消除“只有离店提醒能配置、搬入提醒不能配置”的明显不一致。
- 风险：
  - 没有解决 `离店提醒` 与 `合同到期` 的语义混淆。
  - 没有解决“窗口型提醒”和“客观状态信号”被混成一层的问题。
  - 没有解决提醒链路双实现与遗留旧键的问题，只是继续在旧结构上打补丁。

### 方案 B
- 做法：
  - 保留当前 4 个提醒入口，但重新完成提醒语义定性：
    - `空房查询`：保留，归类为房态运营信号
    - `离店提醒`：保留，归类为合同即将到期窗口提醒
    - `30天搬入`：保留但重命名为“待入住合同”或“即将生效合同”，归类为合同即将生效窗口提醒
    - `合同到期`：保留但重命名为“已到期合同”或“逾期合同”，归类为客观异常待处理事项
  - Settings 只承接“窗口型提醒”的天数配置：
    - 已存在：`contractExpiryAlertDays`
    - 可新增：`upcomingMoveInAlertDays`
  - 明确不把以下项作为本次正式设置：
    - `billReminderDays`
    - `readingReminderDays`
    - `enableNotifications`
  - 后续若进入实现，顺带统一提醒装配路径，优先收口为单一实现真相源。
- 优点：
  - 最符合当前 issue 的真实目标：先把提醒模块的业务语义、配置边界和实现真相源梳理清楚。
  - 四个提醒都保留其业务价值，但不再以模糊命名误导用户。
  - 能自然承接 `fix_006` 的“只让真实生效配置进入设置页”原则。
- 风险：
  - 需要接受一个现实：不是所有“提醒相关字段”都应该进入设置页。
  - 后续进入实现时，可能需要改动 API 标题、前端文案、设置项命名和少量遗留代码清理。

## 6. 推荐方案
- 推荐原因：
  - 推荐采用 **方案 B 的“语义重分层 + 配置边界收口”方案**。
  - 当前问题的本质不是“提醒卡片数量不够”或“提醒窗口没做成设置”，而是“不同类型的提醒被混成同一层，并且只有部分完成了配置化”。
  - 四个提醒本身都有保留价值，但必须先澄清：
    - 哪些是客观状态
    - 哪些是窗口型经营提醒
    - 哪些可以配置
    - 哪些不应配置
- 实施边界：
  - 本次 fix 的后续 `/spec` 若进入实现，建议只覆盖以下边界：
    - 明确四个提醒的正式业务定义和页面命名
    - 把 `30天搬入` 收口为统一窗口语义，并评估新增 `upcomingMoveInAlertDays`
    - 保持 `contractExpiryAlertDays` 为“离店提醒/合同即将到期”唯一真相源
    - 保持 `合同到期` 为“已到期未处理合同”客观状态，不引入可调窗口
    - Settings 只开放已真实进入主链的提醒配置
    - 评估并收口旧版 `getDashboardAlerts()` 等重复装配路径，避免继续双重真相
  - 建议的功能定性：
    - `空房查询`：合理，保留；不进入设置页
    - `离店提醒`：合理，保留；继续使用 `contractExpiryAlertDays`
    - `30天搬入`：合理，保留但应重命名并配置化；可进入设置页
    - `合同到期`：合理，保留但应改名为“已到期合同/逾期合同”；不进入设置页
  - 建议新增但不强行并入本次 fix 的候选提醒：
    - `逾期账单待处理`，因为它属于客观待处理事项，比抽象的 `billReminderDays` 更稳定、更符合经营动作语义
- 明确不在本次修复范围内的内容：
  - 不在本次 fix 中顺手实现完整通知中心或消息投递系统
  - 不在本次 fix 中把 `billReminderDays` 直接暴露为正式设置项
  - 不在本次 fix 中把 `readingReminderDays`、`enableNotifications` 等未成型能力包装成已生效配置
  - 不在本次 fix 中重做整个 Dashboard 布局或视觉风格
  - 不在本次 fix 中扩展新的复杂提醒编排、任务中心或定时任务系统

## 7. 数据修复策略
- 是否需要修历史数据：不需要
- 若需要，修复范围：
  - 若后续引入 `upcomingMoveInAlertDays`，只涉及全局设置默认值与设置页入口补齐，不涉及历史合同/房间/账单事实修复。
  - 若后续清理 `reminderDays` 等旧键，可保留数据库记录作为兼容项，但不应继续在正式页面暴露。
- 若不需要，原因：
  - 当前问题不在于历史业务事实错误，而在于提醒模块的业务定性、命名、配置边界和实现真相源没有完全收口。

## 8. 验收标准
- Dashboard 四个提醒都能被清楚解释为稳定、互不冲突的业务语义，不再出现“看起来相似、实际含义不同”的模糊命名。
- `离店提醒` 与 `合同到期` 必须在页面或文案上清楚区分：
  - 一个是“窗口内即将到期”
  - 一个是“已到期未处理”
- `30天搬入` 若保留为窗口型提醒，窗口天数必须与正式设置项保持一致；若尚未配置化，则不得继续让设置页出现对应正式配置入口。
- Settings 中的提醒类配置只保留真实进入主链的项：
  - 已生效：`contractExpiryAlertDays`
  - 后续若接入：`upcomingMoveInAlertDays`
- `billReminderDays`、`readingReminderDays`、`enableNotifications` 等未形成完整提醒主链的字段，不得继续作为正式生效配置暴露给用户。
- 若进入实现，提醒装配路径应尽量收口为单一真相源，不再长期并存“当前首页实现”和“旧版聚合实现”两套逻辑。

## 9. 回滚条件
- 若后续实现阶段出现以下任一情况，必须立即回滚：
  - 原本已生效的 `contractExpiryAlertDays` 被破坏，导致合同列表、详情页和 Dashboard 离店提醒口径再次不一致
  - `合同到期` 被错误改造成窗口型提醒，导致已逾期合同无法被单独识别
  - `30天搬入` 配置化后出现首页显示与 API 实际筛选窗口不一致
  - 设置页再次暴露未接线提醒项，重新制造“配置存在但不生效”的问题
- 若后续验证发现：
  - 当前首页仍依赖旧版 `getDashboardAlerts()` 的隐藏消费链
  - 某些遗留提醒字段在其他未审计入口被真实使用
  - 新命名会影响既有操作认知或验收脚本
  则必须先补充更细的链路审计与命名方案，再进入实现，而不能直接改文案或直接加设置项。
