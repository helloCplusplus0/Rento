# Fix 011 Analysis - 其他账单类型结构化条目名收口方案

## 1. 问题摘要
- 对应问题：`fix_011`
- 问题级别：`P1`
- 是否阻断修复：是

本问题不是“账单类型模型错误”，而是 `OTHER` 账单缺少结构化条目名承载位，导致真实收费条目在展示层被统一折叠成“其他/其他费用”。  
在当前真实场景下，合同自动生成的钥匙押金、卫生费，以及手动新增的临时收费，虽然底层都应继续归类为 `OTHER`，但前端若只展示“其他”，会直接损害账单可解释性与用户理解成本。

## 2. 根因结论
- 根因一：当前 `Bill` 模型没有承载 `OTHER` 条目名的专用字段，合同自动生成、手动新增、账单编辑都只能把具体条目语义塞进 `remarks`，展示层无法稳定提取“钥匙押金”“卫生费”等具体收费项。
- 根因二：展示层对 `OTHER` 的表达已经分裂：有的页面显示“其他”，有的页面显示“其他费用”，有的页面根本不展示类型，仅用统一 `￥` 图标，导致同一类账单在不同入口缺少统一、可解释的视觉语义。
- 根因三：当前合同自动生成链路中的“清洁费”命名仍沿用历史口径，未与最新业务语义“卫生费”同步；若继续依赖备注文本和页面硬编码展示，不仅会维持命名漂移，也会让未来历史兼容与图标映射更加脆弱。

## 3. 证据链
- 页面预展示链路：
  - 账单列表页的桌面卡片与移动卡片都不直接展示 `bill.type`，卡片左上角统一使用 `￥` 图标块，无法区分租金、押金、其他费用。
  - 账单详情页基础信息对 `OTHER` 固定显示“其他”，不会从现有 `remarks` 中稳定提炼“钥匙押金”“清洁费/卫生费”等具体条目名。
  - 新增账单页的类型选择器对 `OTHER` 使用“其他费用”与 `Plus` 图标，说明展示端已经隐含承认 `OTHER` 需要单独表达，但该口径未同步到账单列表、详情页和合同账单预览。
  - 合同创建预览中，`OTHER` 预览项虽然分别使用了 `Key`、`Sparkles` 图标，但类型 badge 仍只显示“其他”，存在“图标知道是什么、文案却丢失语义”的分裂。
- 服务端实际创建链路：
  - 合同创建后自动生成账单时，钥匙押金与清洁费都以 `Bill.type='OTHER'` 落库，具体条目仅写入 `remarks`，例如“钥匙押金 - 合同...”“清洁费 - 合同...”。
  - 手动新增账单页和编辑账单页也都只有 `remarks`，没有“条目/别名”这样的结构化输入位。
  - 续租链路复用同一套自动生成逻辑，因此会继续制造同样的 `OTHER + remarks` 数据形态。
- 数据模型映射：
  - `Bill` 当前只有 `type / amount / period / remarks / metadata` 等通用字段，没有 `itemLabel`、`displayAlias`、`otherLabel` 这类专用承载位。
  - `metadata` 当前主要服务于水电账单聚合与明细兼容，不适合作为 `OTHER` 条目名的默认承载位。
  - `Contract` 当前只有 `keyDeposit`、`cleaningFee` 两个一次性费用源字段，说明“钥匙押金/卫生费”在合同链路中是明确业务语义，但落到账单模型后丢失为统一 `OTHER`。
- 关键代码位置：
  - Prisma 模型：`prisma/schema.prisma`
  - 合同自动生成账单：`src/lib/auto-bill-generator.ts`
  - 合同创建表单：`src/components/business/ContractForm.tsx`
  - 合同账单预览：`src/components/business/ContractBillPreview.tsx`
  - 新增账单页与表单：`src/components/pages/CreateBillPage.tsx`、`src/components/business/BillInfoForm.tsx`
  - 编辑账单页：`src/components/pages/EditBillPage.tsx`
  - 账单详情展示：`src/components/business/BillBasicInfo.tsx`
  - 账单列表卡片：`src/components/business/bill-card.tsx`、`src/components/business/BillCardCompact.tsx`
- 是否存在历史脏数据：存在较高概率的“展示语义脏数据”。
  - 历史 `OTHER` 账单大多只有 `remarks`，没有结构化条目名。
  - 对于自动生成的“钥匙押金/清洁费”账单，仍可较高置信度从既有 `remarks` 回填条目名。
  - 对于手工新增且备注不规范的历史 `OTHER` 账单，可能无法精准恢复条目名。

## 4. 影响面分析
- 新创建合同：
  - 会直接受影响。
  - 当前自动生成的钥匙押金、清洁费账单虽然事实正确，但展示端只显示“其他”，用户难以判断实际收费项。
- 续租：
  - 会直接受影响。
  - 续租合同复用同一套合同字段与自动生成逻辑，若不补结构化别名，新增续租账单仍会继续丢失展示语义。
- 手工生成账单：
  - 会直接受影响。
  - 新增账单与编辑账单缺少 `OTHER` 别名入口，运营侧只能把具体条目混写进备注，后续展示、搜索和图标映射都不稳定。
- 历史已生成账单：
  - 已存在展示语义缺失问题。
  - 若后续引入结构化条目名字段，历史账单至少需要一轮“可安全回填的自动补录 + 无法识别的人工兜底”策略。
- 统计 / 仪表盘：
  - 当前 fix 不应改变统计主口径。
  - 无论展示成“其他-钥匙押金”还是“其他-卫生费”，统计和筛选的底层正式类型仍应保持 `OTHER`，不能借此衍生新的账单类型枚举或新的统计口径。

## 5. 候选方案对比
### 方案 A
- 做法：
  - 保持当前模型不变，继续只用 `remarks` 承载条目语义。
  - 展示层通过文本规则从 `remarks` 中猜测条目名，例如识别“钥匙押金 - 合同...”“清洁费 - 合同...”后展示成“其他-钥匙押金”“其他-卫生费”。
  - 图标也根据 `remarks` 模糊匹配决定。
- 优点：
  - 改动量最小，不需要 Prisma schema 变更。
  - 对未来数据能快速止血，短期可见效果快。
- 风险：
  - 仍然没有结构化真相源，条目名、图标、编辑入口都会继续依赖文本猜测。
  - 手工创建与历史数据的自由备注很难保证稳定解析，容易继续形成双重真相。
  - 无法真正满足“新增账单、编辑账单提供条目名入口”的诉求。

### 方案 B
- 做法：
  - 保持底层 `Bill.type` 继续使用 `OTHER`，不拆新类型。
  - 为 `Bill` 增加一个仅服务展示语义的可选结构化字段，例如 `itemLabel` 或等价命名，用于承载 `OTHER` 的具体条目名。
  - 合同自动生成链路在创建钥匙押金、卫生费账单时同时写入默认条目名。
  - 新增账单页、编辑账单页仅在 `type='OTHER'` 时显示“条目名”输入入口。
  - 列表页、详情页、催缴汇总页、合同预览页统一使用“正式类型 + 条目名”的展示与图标映射，例如 `其他-钥匙押金`、`其他-卫生费`。
  - 对历史账单做安全回填：优先回填自动生成且 `remarks` 可稳定识别的 `OTHER` 账单；无法可靠识别的保持空值并使用兼容展示降级。
- 优点：
  - 既保持 `OTHER` 作为兜底正式类型不变，又补齐了真实业务所需的展示条目语义。
  - 新增/编辑/自动生成/展示入口都可以围绕同一结构化字段工作，能形成单一真相源。
  - 图标、标签、账单详情、合同预览可以统一映射，用户理解成本最低。
- 风险：
  - 需要一次 Prisma schema 变更与轻量历史数据回填。
  - 实现面会同时触及合同自动生成、手动创建、编辑页、多个展示页，需要边界控制，防止顺手扩张为“重做账单类型体系”。

## 6. 推荐方案
- 推荐原因：
  - 推荐采用 **方案 B 的“保持正式类型不变 + 新增结构化条目名字段”**。
  - `fix_011` 的核心不是让 `OTHER` 变成多个正式账单类型，而是为 `OTHER` 补一个前后端共享、职责清晰的结构化条目名真相源。
  - 该字段的职责应明确冻结为：
    - `Bill.type` 负责正式账单分类
    - `Bill.itemLabel` 负责 `OTHER` 的具体收费条目名
    - `Bill.remarks` 退回备注职责，不再承担结构化条目语义
  - 如果继续只靠 `remarks` 猜测，当前问题最多只是“表面好看一些”，无法真正关闭“新增入口缺失、编辑入口缺失、图标不统一、历史兼容脆弱”这四类问题。
- 实施边界：
  - 保持 `Bill.type` 不变，`RENT / DEPOSIT / UTILITIES / OTHER` 主口径不变。
  - 新增 `OTHER` 的结构化条目名字段，仅用于展示和输入，不参与支付周期、账单状态机、统计主口径和结算计算。
  - 合同自动生成的“清洁费”默认展示名调整为“卫生费”，但当前合同底层字段 `cleaningFee` 可暂不改名，只在展示与生成备注/别名层收口。
  - 新增账单页和编辑账单页只在 `type='OTHER'` 时展示条目名输入入口，其他类型不引入额外复杂度。
  - 账单列表页、账单详情页、合同账单预览页、`fix_010` 的本次应催缴汇总页统一复用同一套 `OTHER` 条目名展示与图标映射。
  - 图标映射应优先基于“结构化条目名 -> 展示图标”，不能再由页面各自手写或继续统一使用 `￥` 占位图标。
- 明确不在本次修复范围内的内容：
  - 不把 `OTHER` 拆成新的正式 `Bill.type` 枚举。
  - 不在当前 fix 中重写账单统计、筛选、聚合和结算主逻辑。
  - 不在当前 fix 中重构 `Contract.cleaningFee` 字段命名或牵动续租/退租计算链的字段层重命名。
  - 不在当前 fix 中要求历史所有 `OTHER` 账单都精准恢复别名；无法可靠识别的历史记录允许降级显示。
  - 不处理 `/contracts/new` 占位页与 `/add/contract` 实际入口的全局路由治理，该问题可作为独立治理项后置。

## 7. 数据修复策略
- 是否需要修历史数据：需要，但以“安全回填 + 降级兼容”为原则。
- 若需要，修复范围：
  - 优先回填以下安全范围内的历史 `OTHER` 账单：
    - 合同自动生成的钥匙押金账单，`remarks` 可稳定识别“钥匙押金”
    - 合同自动生成的清洁费账单，统一回填条目名为“卫生费”
    - 其他 `remarks` 明确、格式稳定的历史 `OTHER` 账单
  - 无法可靠识别的历史手工账单不强行回填，只保留空条目名并在展示层降级为“其他”或“其他-备注摘要”。
- 若不需要，原因：
  - 不适用。
  - 若完全不处理历史数据，列表页、详情页与真实账单库中的大量既有 `OTHER` 账单仍会继续丢失收费条目语义，无法关闭当前 fix。

## 8. 验收标准
- 合同创建后自动生成的 `OTHER` 账单在列表页、详情页、合同预览页中可明确展示为：
  - `其他-钥匙押金`
  - `其他-卫生费`
- 新增账单页与编辑账单页在 `type='OTHER'` 时提供“条目名”输入入口；非 `OTHER` 类型不出现该入口。
- 账单列表页不再统一使用 `￥` 作为所有账单的占位图标，至少能按账单正式类型与 `OTHER` 别名呈现可解释图标。
- `fix_010` 的“本次应催缴汇总”中，若纳入 `OTHER` 账单，也应复用同一展示别名与图标口径，不再退化成单纯“其他”。
- 统计、筛选、状态展示仍继续以底层 `Bill.type='OTHER'` 为正式口径，不因别名引入新的正式类型统计。
- 对历史自动生成的钥匙押金/卫生费账单，至少能完成安全范围内的结构化条目名回填；无法可靠识别的历史手工账单不会被错误批量改写。

## 9. 回滚条件
- 若结构化别名接入后出现以下任一情况，必须停止发布并回滚：
- 若结构化条目名字段接入后出现以下任一情况，必须停止发布并回滚：
  - `OTHER` 账单在列表页、详情页、编辑页、催缴汇总页之间再次出现展示口径分叉
  - 图标映射与账单事实错位，导致押金/租金/其他费用被误识别
  - 新增或编辑 `OTHER` 账单时，条目名字段影响到了底层 `Bill.type`、账单状态或结算逻辑
- 若历史数据回填试运行发现：
  - 大量手工 `OTHER` 账单无法可靠识别条目名
  - 自动回填会误改已支付、已完成或历史备注具有审计意义的账单
  - 回填范围超出安全边界
  则必须降级为“只修未来数据 + 历史数据展示兼容解析”，不得强行批量改写所有历史 `OTHER` 账单。
