# Fix 003 Analysis - 换表后抄表重复告警与计费缺项问题

## 1. 问题摘要
- 对应问题：`fix_003`
- 问题级别：`P1 / fix-now`
- 是否阻断修复：是

本问题不是单一页面提示错误，而是“仪表换表后的抄表主链语义”存在多处不一致：
- 第一类：房间详情页当前接受的换表动作实际上是“停用旧表 + 新增新表”，但系统没有把这条路径冻结成结构化业务语义，前后端仍混用“停用 / 移除 / 解绑 / 删除”表达。
- 第二类：抄表提交 API 的“同日重复抄表”校验实现错误，未围绕“同一仪表 + 提交日期 + 记录类型”做精确判断，导致换表后继续抄表时出现误告警，并把部分仪表排除在本次聚合账单之外。
- 第三类：批量抄表页复用了房间全量仪表列表，但没有像合同详情页抄表弹窗那样按 `isActive` 过滤，导致禁用仪表仍可被录入和计费，形成入口间语义不一致。
- 第四类：`MeterReading` 当前没有结构化区分“合同初始底数”“正常抄表”“退租最终抄表”等记录类型，历史页和重复门禁只能依赖 `remarks` 文本猜测，长期会继续污染抄表和计费语义。
- 第五类：抄表历史页暴露了删除按钮，但查询层硬编码禁止删除，形成“入口看似可删、实际永远失败”的双重真相。

## 2. 根因结论
- 根因一：抄表 API 的重复校验只按“服务器今天”查记录，而不是按用户提交的 `readingDate` 判断；同时它先查当天全局最新 `1` 条记录，再在内存中按 `meterId` 过滤，逻辑并不能正确表达“该仪表该日是否已有正式抄表”。
- 根因二：批量抄表页复用了 `/api/rooms?includeMeters=true` 的房间全量仪表数据，但页面没有像合同详情页抄表弹窗那样过滤 `isActive`，导致禁用仪表仍然参与显示、录入与费用计算。
- 根因三：`MeterReading` 缺少结构化 `recordType / sourceType` 语义，合同创建时写入的“初始底数”、正常抄表、退租最终抄表都落在同一模型与同一重复门禁之下，只能靠 `usage=0` 与 `remarks` 文本启发式区分。
- 根因四：房间仪表管理的“移除”文案声称是“解绑且保留历史”，但后端实际只有“有历史则停用、无历史则硬删除”，没有真实解绑字段或换表专用流程，导致前端话术、用户心智和后端实际行为不一致。
- 根因五：抄表历史删除链路前后不一致。历史页前端与 API 都表现为“未出账可删除”，但查询层统一禁止删除，用户看到的是功能存在但永远无法落地的伪能力。
- 根因六：数据库层当前只有普通索引，且开发验证库中已经存在历史重复组；如果在未清洗历史数据前强行落数据库级唯一约束，会直接阻断现有环境，因此当前阶段只能冻结为“应用层精确门禁 + 历史重复审计”的兼容路径。

## 3. 证据链
- 页面预展示链路：
  - 合同详情页抄表弹窗从 `/api/rooms/[roomId]/meters` 拉取房间全部仪表，再在前端只保留 `isActive=true` 的仪表展示给用户，见 `src/components/business/SingleMeterReadingModal.tsx`。
  - 用户输入后，页面会正确基于 `meter.lastReading` 计算用量与费用，因此换表后“新表默认底数为 0、展示用量和费用正常”符合当前前端逻辑。
  - 用户截图证据表明：同一次提交中有 `2` 个仪表成功入库，同时出现“今日已存在该仪表的抄表记录”告警，并只生成 `1` 个聚合账单，说明系统走的是“部分成功 + 部分跳过”的提交路径，而不是前端展示层计算错误。
  - 主页快捷操作也提供了 `批量抄表` 入口，该入口最终进入 `src/components/pages/BatchMeterReadingPage.tsx`。
  - 批量抄表页通过 `/api/rooms?includeMeters=true` 读取房间及仪表列表；修复前它会把 `A101` 的 `5` 个仪表全部展示出来，包含已禁用的 `冷水1`，与合同详情页抄表弹窗只展示 `4` 个激活仪表的表现不一致。
- 服务端实际创建链路：
  - `POST /api/meter-readings` 在逐条处理每个读数时，会先执行“今日是否重复”校验，再决定是否创建读数，见 `src/app/api/meter-readings/route.ts`。
  - 当前重复校验固定使用服务器当天的 `todayStart/todayEnd`，且调用 `meterReadingQueries.findAll({ limit: 1, startDate, endDate })` 后再在内存中 `find(r => r.meterId === readingData.meterId)`，这并不是“按仪表 + 按提交日期”的数据库侧过滤。
  - 一旦某个仪表被判定重复，该条记录会被 `warnings.push(...); continue` 跳过，因此不会进入 `results`，后续 `groupReadingsByContract(results)` 生成的聚合账单自然会缺少这一仪表项目。
  - 合同创建时会自动向 `meter_readings` 写入 `usage=0 / remarks='合同创建时的仪表底数'` 的记录，见 `src/app/api/contracts/route.ts`；退租时也会写最终抄表记录，说明“非正常抄表记录”本就存在于同一张表中。
- 数据模型映射：
  - `Meter` 目前只有 `roomId + isActive` 两个当前关联状态字段，没有解绑历史或换表专用语义字段，见 `prisma/schema.prisma`。
  - 修复后 `MeterReading` 已具备结构化 `recordType` 字段，但当前开发验证库因历史重复组仍未强制落数据库级唯一约束，边界冻结为“应用层精确门禁 + 历史重复审计清单”，见 `prisma/schema.prisma`、`src/lib/queries.ts` 与 `src/app/api/meter-readings/route.ts`。
  - 抄表历史页在修复后优先使用结构化 `recordType` 做展示与筛选，仅对未知/存量坏数据做兼容兜底，不再依赖备注文本作为主真相源，见 `src/components/pages/MeterReadingHistoryPage.tsx`。
  - Context7 的 Prisma 文档明确建议：需要稳定表达复合唯一性时，应通过 `@@unique([...])` 建立数据库约束；需要保留历史事实时，应使用显式 referential actions（如 `Restrict`/`NoAction`）而不是依赖级联删除。
- 关键代码位置：
  - 合同抄表弹窗：`src/components/business/SingleMeterReadingModal.tsx`
  - 批量/合同共用抄表提交 API：`src/app/api/meter-readings/route.ts`
  - 批量抄表页：`src/components/pages/BatchMeterReadingPage.tsx`
  - 主页快捷入口：`src/components/business/dashboard-home.tsx`
  - 抄表查询层：`src/lib/queries.ts`
  - 合同创建初始底数写入：`src/app/api/contracts/route.ts`
  - 房间仪表获取与新增：`src/app/api/rooms/[id]/meters/route.ts`
  - 仪表移除/停用：`src/app/api/meters/[meterId]/route.ts`
  - 抄表历史页与删除按钮：`src/components/pages/MeterReadingHistoryPage.tsx`
  - 抄表记录详情删除 API：`src/app/api/meter-readings/[id]/route.ts`
- 是否存在历史脏数据：存在高概率。
  - 可能已经存在“同一仪表同日重复正式抄表”或“应允许入账的换表后新表读数被误跳过”的历史情况。
  - 历史页对“初始底数 / 正常抄表”的展示筛选完全依赖备注文本，旧数据若备注不规范，将继续被误分类。
  - 历史上通过 UI 尝试删除抄表记录的用户，已经暴露过一个“假删除入口”的治理问题。

## 4. 影响面分析
- 新创建合同：
  - 合同创建当天会写入初始底数记录；若当天立即做正式抄表，现有重复门禁与文本型历史分类存在天然冲突。
- 续租：
  - 若续租流程未来继续复用“写初始底数 / 接续抄表”的模型而不区分记录类型，问题会原样继承。
- 手工生成账单：
  - 抄表 API 的 `results` 一旦缺少某个仪表，聚合账单就会缺项，导致水电费金额低于真实应收。
- 批量抄表入口：
  - 若批量抄表页不按 `isActive` 过滤仪表，会直接让禁用表继续参与录入与计费，导致它与合同详情页抄表弹窗出现双重真相。
- 历史已生成账单：
  - 受影响的不只是换表当天；任何“同日存在其他结构化/非结构化读数”的仪表，都可能被当前重复门禁误伤，留下缺少某类仪表费用的历史聚合账单。
- 统计 / 仪表盘：
  - 抄表历史页、未来任何按记录类型统计的页面，都会继续受 `remarks` 文本猜测逻辑影响。
- 仪表管理：
  - 当前“移除 = 解绑”的 UI 表达会继续误导用户；实际上系统只是在做停用或硬删除，用户会把换表路径理解为一种不存在的结构化解绑流程。

## 5. 候选方案对比
### 方案 A
- 做法：
  - 仅修复 `POST /api/meter-readings` 的重复校验，把当前“按今天 + limit 1 + 内存过滤”改成“按 `meterId + readingDate` 精确查询”。
  - 保留 `remarks` 启发式区分“初始底数/正常抄表”的方式不变。
  - 保留仪表“移除”与历史删除入口的现状，只做文案微调。
- 优点：
  - 改动最小，最可能快速消除截图中的误告警与聚合账单缺项。
  - 不涉及 Prisma schema 变更，实施风险较低。
- 风险：
  - 仍未结构化区分“初始底数 / 正常抄表 / 退租抄表”，同日业务语义仍依赖备注文本，后续问题还会复发。
  - 历史页删除入口、仪表移除语义双重真相继续存在。
  - 缺少数据库级唯一约束，仍有并发重复写入风险。

### 方案 B
- 做法：
  - 冻结当前可接受的换表路径：`停用旧表 + 新增新表`，旧表保留为停用状态，不在本次 fix 中扩展成完整解绑模型。
  - 为 `MeterReading` 引入结构化记录类型，例如：
    - `INITIAL_BASELINE`
    - `REGULAR_READING`
    - `CHECKOUT_FINAL`
  - 重写重复校验规则：
    - 仅对 `REGULAR_READING` 施加“同一仪表 + 同一业务日期 + 同一记录类型”的唯一性门禁
    - 校验必须基于提交的 `readingDate`
    - 查询必须在数据库侧按 `meterId` 精确过滤，而不是先全量/分页后内存判断
  - 同步收口批量抄表页：
    - 与合同详情页抄表弹窗保持一致，只展示 `isActive=true` 的激活仪表
    - “提交抄表”按钮状态只由 React 状态流驱动，不再混用 DOM 直改
  - 数据库边界按真实环境冻结为：
    - 当前阶段先采用“应用层精确门禁 + 历史重复审计”
    - 不在未清洗历史重复数据前强行落数据库级唯一约束
  - 同步收口历史页与删除能力：
    - 若当前阶段不允许物理删除，则移除历史页删除按钮或显式改为“暂不支持”
    - 若允许有限删除，则只允许删除 `REGULAR_READING` 且 `isBilled=false` 的记录，并同步更新查询层与 API 门禁
  - 同步修正文案：把房间页“移除”解释回当前真实行为，明确“当前是停用保留历史，不是物理解绑”。
- 优点：
  - 能同时修复当前截图问题、同日初始底数冲突、历史页误分类和删除入口双重真相。
  - 更符合项目“单一真相源、历史事实优先保留”的治理要求。
  - 后续若继续做换表、抄表统计、异常审计，也能直接复用这次冻结的结构化语义。
- 风险：
  - 需要触及 Prisma schema、抄表 API、合同创建、退租抄表、历史页与部分 UI 文案。
  - 若直接加数据库唯一约束，必须先评估历史重复数据并给出迁移前清洗策略。

## 6. 推荐方案
- 推荐原因：
  - 推荐采用 **方案 B 的“结构化抄表语义 + 精确重复门禁 + UI 文案收口”**，但严格控制在当前 `fix_003` 的最小业务边界内。
  - 当前问题已不只是一个 `if` 判断写错，而是“抄表记录类型、换表路径和历史删除能力”三者都存在双重真相。
  - 若只修 API 的重复判断，短期能缓解截图问题，但“合同初始底数 / 正常抄表 / 退租最终抄表”继续共用一套无类型语义，后续很容易在别的入口再次出问题。
- 实施边界：
  - 冻结换表业务边界：
    - 当前接受的换表动作固定为：停用旧表、保留展示、再新增新表
    - 本次 fix 不实现真正的物理解绑历史模型
  - 冻结批量抄表边界：
    - 主页快捷操作进入的 `批量抄表` 入口必须与合同详情页抄表弹窗遵守同一套激活仪表过滤语义
    - 批量抄表页的“提交抄表”按钮只允许由声明式状态流控制：无输入禁用、有有效输入启用、存在错误重新禁用
  - 冻结抄表记录边界：
    - `MeterReading` 必须具备结构化 `recordType`
    - 合同创建底数、正常抄表、退租最终抄表必须写入明确类型
  - 冻结重复校验边界：
    - 只阻止“同一仪表在同一业务日期重复提交正式抄表”
    - 不应把 `INITIAL_BASELINE` 误判为 `REGULAR_READING` 的重复
    - 不应使用服务器“今天”代替用户提交的 `readingDate`
  - 冻结历史页边界：
    - 记录类型展示必须优先依赖结构化字段，而不是 `remarks` 文本
    - 删除入口和服务端门禁必须口径一致，不能继续保留伪能力
  - 冻结数据库边界：
    - 当前开发验证库先采用“应用层精确门禁 + 历史重复审计清单”的兼容路径
    - 数据库级唯一约束需在后续历史重复数据清洗完成后再独立推进
    - 对保留历史事实的关系，评估是否需要从 `Cascade` 调整为更保守的 referential action
- 明确不在本次修复范围内的内容：
  - 不在本次 fix 中重构完整的仪表绑定历史表或资产履历系统。
  - 不在本次 fix 中顺手重做整个房间仪表管理 UI。
  - 不在本次 fix 中扩展成“同一仪表一天多次正式抄表”的复杂补录策略；若未来需要，应另起 fix 或 plan 单独设计。

## 7. 数据修复策略
- 是否需要修历史数据：需要，但分层处理。
- 若需要，修复范围：
  - 结构化类型回填：
    - 为历史 `MeterReading` 评估回填 `recordType`
    - 可优先使用安全启发式：
      - `usage = 0` 且备注含“合同创建 / 初始读数 / 底数” -> `INITIAL_BASELINE`
      - 退租流程写入的最终抄表 -> `CHECKOUT_FINAL`
      - 其余已有正常计费用途记录 -> `REGULAR_READING`
  - 重复数据审计：
    - 审计“同一 `meterId` + 同一自然日 + 多条可能属于 `REGULAR_READING`”的历史记录
    - 仅输出审计清单，不默认自动删除或合并
    - 当前开发验证库已确认存在历史重复组，因此本次 fix 不强行追加数据库级唯一约束
  - 缺项账单审计：
    - 对换表后当日提交且只生成部分聚合账单明细的历史场景，优先输出审计清单，人工确认是否需补账单
- 若不需要，原因：
  - 不适用。本问题会影响真实计费事实，不能只修未来路径而忽略历史审计。

## 8. 验收标准
- 在“停用旧表 + 新增新表”的换表路径下，合同详情页再次抄表时，新表提交不会被误判为“今日已存在该仪表抄表记录”。
- 同一次提交中，所有有效仪表都会进入 `results`，生成的聚合水电账单不会遗漏应纳入的电表项目。
- 抄表重复校验必须基于提交的 `readingDate` 与 `meterId`，不能继续使用服务器当前日期替代。
- 合同创建初始底数、正常抄表、退租最终抄表在数据层有明确记录类型，不再依赖 `remarks` 文本猜测。
- 抄表历史页对“初始底数 / 正常抄表”展示过滤必须复用结构化字段，不再依赖备注关键字。
- 历史页删除入口与服务端策略保持一致：
  - 若当前阶段禁止删除，则页面不应继续展示可执行删除按钮
  - 若允许删除，则删除门禁必须与查询层、API 一致
- 房间详情页“移除仪表”相关文案必须准确反映当前真实行为，不再把“停用保留历史”表述成“已解绑”。
- 主页快捷操作 -> `批量抄表` 入口必须可达，且批量抄表页只展示激活中的仪表；对 `A101` 而言，应只展示 `4` 个激活仪表，不再显示禁用的 `冷水1`
- 批量抄表页“提交抄表”按钮必须满足：
  - 无输入时禁用
  - 有有效输入时启用
  - 存在校验错误时重新禁用
- 至少补一条主链验证路径：
  - 合同生效中 -> 抄表成功 -> 停用旧表 -> 新增新表 -> 再次抄表 -> 电表费用正确进入聚合账单
- 至少补一条同日验证路径：
  - 合同创建当天存在初始底数 -> 当天执行正式抄表 -> 不会因初始底数记录而被误判重复

## 9. 回滚条件
- 若结构化 `recordType` 引入后出现以下任一情况，必须立即回滚：
  - 合同创建底数无法写入
  - 正常抄表无法生成账单
  - 退租最终抄表被错误分类或丢失
- 若重复校验收口后出现以下任一情况，必须立即回滚：
  - 同一仪表同一日期可重复写入多条正式抄表且未被门禁阻止
  - 正常换表路径仍然出现误告警并导致聚合账单缺项
- 若历史数据回填或迁移前审计发现：
  - 历史重复记录无法安全判定主次
  - 自动修复会破坏已生成账单与抄表事实的可追溯性
  则必须降级为“只修未来路径 + 导出历史审计清单”，不得强行自动清洗历史数据。
