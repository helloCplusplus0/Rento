## 5. 预期结果
- 合同详情页“抄表录入”弹窗只展示激活中的仪表；对 `A101` 而言，应展示 `电1 / 冷水2 / 冷水12 / 热水1`，不再展示禁用的 `冷水1`
- 批量抄表页必须与合同详情页保持一致，只展示激活中的仪表；主页快捷操作进入的 `批量抄表` 入口也必须遵守同一套语义
- 换表路径下，禁用旧表 + 新增新表后，再次正式抄表不会再出现“今日已存在该仪表抄表记录”的误告警
- 同一次有效抄表提交中，所有应纳入的激活仪表都必须进入聚合账单，不能再遗漏电费项目
- 抄表历史页应基于结构化记录类型展示，且删除入口与服务端策略一致，不再保留伪能力
- 批量抄表页“提交抄表”按钮应遵守统一状态逻辑：
  - 无输入时禁用
  - 有有效输入时启用
  - 存在校验错误时重新禁用

## 6. 实际结果
- 修复前，合同详情页抄表弹窗已按激活状态过滤仪表，但批量抄表页仍会展示禁用的 `冷水1`，形成入口间语义不一致
- 修复前，换表后继续抄表时，正式抄表重复门禁会误伤新表或其他有效读数，导致聚合账单缺少应纳入的仪表项目
- 修复前，批量抄表页“提交抄表”按钮还混用 React 状态与 DOM 直改，容易造成按钮状态、回执展示与真实提交流程漂移

## 9. 收口结论
- `fix_003` 已按“停用旧表 + 新增新表”的既定换表路径完成收口，合同详情页抄表弹窗与批量抄表页现在统一只展示激活中的仪表
- 针对 `A101` 的真实样例：
  - 房间详情页保留 `5` 个仪表卡片，其中 `冷水1` 为禁用，其他 `4` 个仪表为激活
  - 合同详情页抄表弹窗只展示 `4` 个激活仪表，这是正确设计
  - 批量抄表页修复后也只展示 `4` 个激活仪表，不再展示禁用的 `冷水1`
- 抄表记录已引入结构化 `recordType`，并打通三条主写入路径：
  - 合同初始底数：`INITIAL_BASELINE`
  - 正常抄表：`REGULAR_READING`
  - 退租最终抄表：`CHECKOUT_FINAL`
- 正式抄表重复门禁已改为基于提交的 `readingDate + meterId + REGULAR_READING` 精确判断，不再使用服务器“今天”替代业务日期，也不会再把初始底数误判为正式抄表重复
- 抄表历史页已按结构化 `recordType` 展示，并移除伪删除能力；房间详情页“移除仪表”文案也已回归真实语义：有历史则停用保留，无历史才可硬删除
- 批量抄表页“提交抄表”按钮逻辑已收口为声明式状态流：
  - 无输入禁用
  - 有有效输入启用
  - 存在校验错误重新禁用
- 数据库边界按当前开发验证库的真实状态冻结为：
  - 已通过兼容路径同步 `recordType` 字段到当前 PostgreSQL
  - 由于历史数据中存在重复组，本次不强行落数据库级唯一约束
  - 当前采用“应用层精确门禁 + 历史重复审计清单”的兼容实现
