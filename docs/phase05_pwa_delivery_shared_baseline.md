# Phase05 PWA Delivery Shared Baseline

## 一、文档目的

本文档用于冻结 `phase05-pwa-delivery-*` 全部子任务共享的边界、允许路线、禁止路线与统一判断标准，避免后续 `/spec` 各自扩写出不同解释。

> 说明：当前默认工作流仍然是“真实场景验证与 fix 闭环”。本基线用于先冻结候选 `phase05` 的统一口径，未经用户审核，不自动切换默认入口。

## 二、共享前提

- `phase01-restart-foundation-*` 已完成
- `phase02-auth-gate-*` 已完成
- `phase03-consistency-hardening-*` 已完成
- `phase04-performance-and-ops-*` 已完成
- `fix_008` 已完成移动端路线分析
- 当前 UI 仍为冻结资产
- 当前数据库主线固定为 PostgreSQL
- 当前开发服务器由用户手动启动，AI 不主动后台运行 `npm run dev`
- 当前目标不是做第二套移动端产品，而是在单一 Web 主线上实现可安装 PWA 交付闭环

## 三、共享判断标准

- 默认优先保持单仓库、单 UI、单业务真相源稳定
- 默认优先安卓优先、受控环境优先，而不是一开始承诺所有设备与所有浏览器
- 默认优先渐进增强：核心业务必须先能作为普通 Web 正常运行，PWA 只是增强层
- 默认优先最小 service worker、最小离线壳、最小更新策略，而不是先追求完整离线
- 默认优先问题可解释、可验证、可回滚，而不是“看起来像原生 App”
- 默认要求实现低复杂度，在满足以上原则前提下保持轻量、高效、灵活、低耦合

## 四、允许路线

- 允许冻结正式支持矩阵、环境分层、安装边界与退化策略
- 允许补 manifest、图标、启动体验、安装提示或安装引导
- 允许实现最小 service worker、最小离线兜底页与更新提示
- 允许审计并修复关键页面的移动端可用性问题
- 允许更新与 PWA、安装和私有部署直接相关的顶层文档、环境模板和发布说明
- 允许补最小真机验收路径与最小调试说明

## 五、禁止路线

- 禁止把 `phase05` 演化为 Flutter / 原生 Android / iOS 双端项目
- 禁止重建一套专属移动端 UI 或第二套路由结构
- 禁止未经支持矩阵冻结就承诺“所有浏览器都支持安装”
- 禁止把完整业务 API 离线缓存、离线数据库、后台同步和推送系统混入本阶段主线
- 禁止为了移动端交付放松当前认证、安全门禁和私有部署要求
- 禁止未经专项判断把缓存逻辑做成业务主链的隐性真相源

## 六、统一治理语义

### 6.1 支持矩阵

- 正式支持优先级默认以 `Android + Chrome` 为第一优先级
- `Android + Edge` 与其他主流 Chromium 系浏览器可作为次级兼容目标，但不默认承诺与 Chrome 同等级安装体验
- iOS 不作为本阶段正式承诺目标，只允许保留基础访问和必要安装说明
- 若浏览器或设备不支持 PWA 安装能力，系统必须自动退化为普通响应式 Web

### 6.2 环境分层

- 本地开发环境：默认禁用 service worker 注册与缓存验证，不把 PWA 行为作为日常页面开发前提
- 受控测试环境：用于真机验证安装、启动、更新、离线兜底与退化路径，可按专项测试启用 service worker
- 私有部署生产环境：必须满足 HTTPS 与受控网络前提，沿用与受控测试环境一致的 PWA 交付与更新口径

### 6.3 退化策略

- 浏览器或设备不支持安装能力时，自动退化为普通响应式 Web，不阻断登录、查询、录入与其他在线主链操作
- service worker 不可用、注册失败、被手动清除或更新异常时，系统必须仍可回到普通 Web 访问路径
- 离线场景只允许提供最小离线兜底页，不把离线能力扩张为可写业务的本地真相源

### 6.4 Service Worker 与缓存

- service worker 必须服务于“安装壳、静态资源、最小离线兜底与更新策略”
- 允许缓存的边界仅限于应用壳静态资源、`manifest`、图标与最小离线兜底资源
- 不默认缓存动态业务接口、鉴权态业务页面响应或其他可能成为业务真相源的动态数据
- 缓存策略必须能解释：
  - 缓存什么
  - 不缓存什么
  - 何时更新
  - 何时失效
  - 出问题如何回滚

### 6.5 安装与更新

- 安装能力必须在私有部署与 HTTPS 前提下设计
- 安装后的启动体验、图标、应用名、更新提示必须口径一致
- 用户必须能理解：
  - 如何安装
  - 如何获取更新
  - 更新失败时会怎样
  - 不支持安装时如何使用普通 Web

### 6.6 移动端页面可用性

- 移动端优化优先解决真实经营场景中的查询、提醒、现场录入与基础操作
- 不以重建桌面页面为前提，而以最小布局收口、safe-area、软键盘处理和导航优化为主
- 安装态与浏览器态的行为差异应尽可能收敛到最小

## 七、固定子任务顺序

- `phase05-pwa-delivery-01-baseline-and-support-matrix-freeze`
- `phase05-pwa-delivery-02-install-shell-and-manifest-hardening`
- `phase05-pwa-delivery-03-service-worker-and-update-strategy`
- `phase05-pwa-delivery-04-mobile-layout-and-key-page-usability-closure`
- `phase05-pwa-delivery-05-private-deployment-and-installation-readiness`

## 八、统一验证要求

- 每个子任务至少执行：
  - `npm run lint`
  - `npm run type-check`
- 若变更涉及安装壳或 manifest，必须证明正式支持浏览器中安装路径可执行
- 若变更涉及 service worker，必须证明：
  - 核心业务在无 SW 或 SW 失效时仍能正常退化为普通 Web
  - 更新策略与缓存策略可解释
  - 至少一条更新验证路径可执行
- 若变更涉及移动端页面可用性，必须证明：
  - 无严重横向溢出
  - 无主要操作被底部导航或软键盘遮挡
  - 关键页面在主流手机尺寸下具备可接受可用性
- 若变更涉及部署与安装说明，必须证明用户能够理解安装、更新与失败退化流程

## 九、阶段停顿门禁

- 当前阶段文档一旦产出完成，必须停止并等待用户审核
- `phase05-pwa-delivery-01-*` 仅用于冻结支持矩阵、环境分层、退化策略、缓存原则与验收口径，不得提前进入安装实现、SW 实现或页面修复
- 未经用户明确指示，不得直接进入 `phase05-pwa-delivery-02-*` 及后续实现子任务
- 后续任何子任务都不得越过本共享基线单独扩张范围
