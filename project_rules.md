# project_rules.md

## 1. 范围与边界
- 当前项目定位为“私有租赁管理后台”，默认服务于自有房源经营，不以开放注册 SaaS 为目标。
- 当前默认优先做真实场景验证、主链问题修复与发布门禁收口，不优先扩张新功能面。
- 所有设计必须围绕真实租务流程：房源、租客、合同、账单、仪表、抄表、退租、续租。

## 2. 安全与发布规则
- 在鉴权落地前，禁止把应用作为公网匿名可访问后台部署。
- 生产环境必须使用私有 `.env`；`.env.example` 仅作为模板，不允许把真实局域网地址、密码、密钥回写到共享文档。
- 任何涉及认证、会话、来源限制、CORS 的调整，都必须同时更新实现、环境模板和文档。
- 新增管理页或调试入口时，必须先判断其是否应受鉴权保护。

## 3. 数据与模型规则
- 数据库主线固定为 PostgreSQL，禁止在当前主路径继续引入 SQLite 作为开发/同步方案。
- `Meter` 必须被视为独立资产；移除房间绑定不等于删除历史数据。
- `MeterReading`、`Bill`、`BillDetail` 的历史记录优先保留，禁止用级联删除掩盖业务历史。
- 涉及账务的金额字段必须保持语义稳定；若有“月租金”“总租金”“首期应收”等概念，必须明确区分，不允许混算后只靠 UI 猜测。
- 删除房间、删除合同、退租、续租等破坏性操作，必须经过业务门禁校验，而不是只看页面状态。

## 4. UI 与交互规则
- 当前 UI 视觉风格、布局体系、组件审美默认冻结，不做无明确收益的重构。
- 允许的 UI 变更仅限：修复明显 bug、减少闪烁、提升移动端可用性、优化信息架构，但不得破坏既有设计语言。
- 新增功能优先复用既有页面模式和组件表达，避免出现另一套设计系统。
- 若需要调整交互，应优先把问题写进文档和计划，再实施最小改动。

## 5. 目录治理规则
- 根目录仅保留当前有效入口：运行配置、环境模板、项目总览和顶层治理文档。
- 历史任务文档迁入 `docs/archive/`；历史性脚本迁入 `scripts/archive/`。
- `src/app` 中非正式业务入口的页面，应明确标注为“开发辅助”或“待归档候选”。
- 运行时产物目录如 `logs/`、`backups/`、`nginx/ssl/`、本地数据库文件，不应继续作为版本控制主路径的一部分。

## 6. 文档规则
- `README.md` 负责对外总览与启动说明，不承载过细的历史任务细节。
- `AGENTS.md` 负责入口摘要与执行总约束。
- `project_rules.md` 负责刚性规则与门禁。
- `architecture_map.md` 负责代码向结构映射与目录说明。
- `plan.md` 负责当前阶段顺序、目标、验收与近期行动计划。
- 阶段级设计文档模板固定为 `docs/phaseX_<workflow>_architecture_plan.md`、`docs/phaseX_<workflow>_dev_plan.md`，必要时补充 `docs/phaseX_<workflow>_shared_baseline.md`。
- 子任务 `spec` 目录模板固定为 `.trae/specs/phaseX-<workflow>-<nn>-<task-name>/`。
- 真实场景验证问题报告模板固定为 `docs/fix/fix_XXX_issue_<topic>.md`，只记录问题事实、复现路径、预期与实际差异，不写实施方案。
- 根因与方案分析文档模板固定为 `docs/fix/fix_XXX_analysis_<topic>.md`，必须明确根因、证据链、影响面、数据修复策略、验收标准与回滚条件。
- `docs/fix/fix_issue_template.md` 与 `docs/fix/fix_analysis_template.md` 分别作为 `issue` 与 `analysis` 的模板真相源。
- 当用户发起 `/plan` 后，必须先同步顶层规范文档，再产出阶段级设计文档；文档产出后必须停下，等待用户审核后才能进入 `/spec`。
- `phase04-performance-and-ops-01-*` 只允许冻结共享边界、页面初始分类口径与验收方向，不允许提前进入查询优化、观测补强或页面门禁实现。
- 归档文档默认只读，不再作为当前实现依据。

## 6.1 Fix 工作流规则
- 新发现的问题必须先进入 `docs/fix/` 的 `issue` 文档；未完成 `analysis` 文档前，禁止直接进入 `/spec`、实现或数据修复。
- `issue` 文档只负责问题事实；`analysis` 文档才负责方案判断与修复边界，禁止在 `issue` 文档中混入直接实施结论。
- fix 闭环默认顺序固定为：`issue -> analysis -> /spec -> 修复 -> 验收 -> 提交并推送`。
- 若 fix 已超出局部修补边界，或涉及多个主链语义同时重定义，必须回到 `/plan`，不得在单个 fix 中扩张成隐性新阶段。

## 7. 代码规则
- 优先保持单一实现路径，发现历史兼容逻辑时要写清原因与退出条件。
- 查询逻辑优先数据库侧过滤、分页与聚合，避免在 API 中先全量查询再内存过滤。
- 对 Prisma schema、迁移脚本、部署脚本的调整，必须保证口径一致；若存在历史遗留，必须在文档中标注，不得伪装成已完成一致性。
- 对显著影响运行边界的路由、脚本、环境变量，必须有注释或文档解释其用途。
- 开发辅助页、性能页、验证页、演示页必须明确分类、用途与门禁边界，不得长期污染正式业务入口。
- fix 修复默认优先最小边界，禁止借修复问题之名顺手扩写无关功能、重构 UI 风格或改动未纳入分析文档的实现范围。
- 任何涉及合同、账单、支付周期、仪表、抄表主链的修复，必须在实施前明确：
  - 是否影响历史数据
  - 是否影响其他生成路径、统计路径或入口
  - 是否需要补数据修复策略
  - 至少一条可执行的主链验证路径
- 在确保实现符合最佳实践的前提下，始终优先保持轻量、高效、灵活、可扩展、低复杂度、低耦合的设计风格。
- 禁止在持续迭代中把系统演化为庞大、臃肿、高心智负担、不可持续推进和不可持续维护的结构。

## 8. 发布门禁
- 提交前至少完成：`npm run lint`、`npm run type-check`。
- 发布前至少完成：容器启动、`/api/health` 可用、核心业务流 smoke test。
- 若变更涉及账单、合同、仪表、抄表任一主链，必须补充最少一条可执行验证路径。

## 9. 当前已知治理债务
- 迁移锁与早期迁移文件仍带有 SQLite 历史痕迹，当前通过部署脚本兼容；在单独任务中再完成正式收口。
- 最小鉴权门禁已落地，但角色控制、最小审计与公网发布所需的完整安全边界仍未完成。
- 开发辅助页面、性能页面、验证页面已完成当前阶段的分类与门禁收口，后续仍需按真实使用价值继续评估其长期保留策略。
- 当前已进入真实场景验证与 fix 闭环阶段；支付周期、账单生成、统计口径等主链问题需通过 `issue -> analysis -> /spec` 的方式逐项收口。
