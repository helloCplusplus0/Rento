# project_skills.md

## 1. 合同锚点技能
- `Contract` 是租务事实主锚点，账单、续租、退租、抄表、房态变化都应优先围绕合同建模。
- 页面交互若与合同主链冲突，应优先修正页面表达，不轻易重写数据模型。
- 删除、编辑、续租、退租等操作必须首先判断对合同链和账务链的影响。

## 2. 多仪表建模技能
- Rento 的核心差异化是“一个房间可绑定多个仪表”。
- 不能退回通用产品的“一房一电一冷一热”假设。
- `Meter` 作为独立资产存在，房间绑定关系服务于当前业务，不应吞掉历史。
- 移除仪表绑定不等于删除历史抄表与历史账单事实。

## 3. 历史保留技能
- 对账单、抄表、合同、仪表历史，优先保留事实记录，而不是通过级联删除制造“系统看起来很干净”的假象。
- 凡是会影响历史追溯的删除逻辑，都应视为高风险改动。
- 若必须移除当前关联，应优先建模为解绑、停用、归档，而不是物理删除。

## 4. 删除门禁技能
- 房间处于 `OCCUPIED` 或 `OVERDUE` 时，不应按普通空房直接删除。
- 合同未结束、账单未结清、仪表仍有关联历史时，删除操作必须被业务门禁拦截。
- 删除门禁应由服务端规则保证，而不是只依赖前端按钮状态。

## 5. 账务语义技能
- 必须区分 `monthlyRent`、`totalRent`、押金、首期应收、账单应收额等不同语义。
- 支付周期（月付、季付、半年付、年付）应明确映射到账单生成策略，不允许 UI 有表达、后端无语义。
- 多仪表账单的聚合逻辑必须可解释、可追溯、可拆回明细。

## 6. UI 稳定技能
- 当前 UI 视为已确认资产，默认不重构视觉风格。
- 允许做的改动：减少闪烁、调整信息组织、修复明显重复入口、提升移动端可用性。
- 新功能优先沿用既有页面模式与组件结构，不新造一套设计语言。

## 7. 公网门禁技能
- 在鉴权落地前，Rento 只适合本地或受控私有环境，不满足公网开放条件。
- 页面与 API 必须统一门禁，不能出现“页面有保护、API 裸露”的假安全。
- 对外部署前至少应具备：登录、会话保护、来源限制、最小审计线索。

## 8. 目录归档技能
- 根目录只保留当前有效入口文档。
- 历史任务文档进 `docs/archive/tasks/`，历史需求与测试草稿进 `docs/archive/legacy/`。
- 历史脚本进 `scripts/archive/`，除非仍然是当前运行链所必需。

## 9. 迁移链治理技能
- 当前主线已是 PostgreSQL，但迁移链仍有 SQLite 时代遗留。
- `migration_lock.toml`、历史 SQL、兼容脚本需要被明确标记为“兼容项”而不是“当前最佳实践”。
- 在未设计迁移链整改方案前，不要贸然修改历史锁文件，以免破坏已有部署路径。

## 10. 重启验收技能
- “达到重启条件”至少包含三层：
- 工程层：`lint`、`type-check`、`build`、`prisma validate` 通过。
- 治理层：顶层文档完整、目录收口、真相源单一、过时入口归档。
- 上线层：鉴权、安全门禁、环境卫生、迁移链一致性达到最低标准。
- 只有前两层满足、第三层未满足时，可判定为“允许恢复开发，但不允许公网发布”。
