# Phase03 开发态门禁运行入口 Spec

## Why
当前开发态手动启动入口没有把最小门禁、数据库连接和验证脚本运行上下文统一收口，导致浏览器验证、脚本验证和手动调试容易落在不同环境里。需要用一个低复杂度、显式、可重复执行的开发启动入口，稳定后续 `phase03` 验证链路。

## What Changes
- 新增一个由用户手动执行的开发启动入口，统一加载或校验开发态所需的认证、数据库和缓存环境变量
- 为开发态提供清晰的失败提示，避免在浏览器登录失败后才发现是运行参数缺失
- 补充轻量的开发便捷入口与文档说明，确保浏览器验证与脚本验证共享同一套上下文
- **BREAKING**：项目默认推荐的本地开发启动方式从裸 `npm run dev` 升级为带完整上下文的手动启动入口

## Impact
- Affected specs: 开发态运行入口、最小门禁可用性、验证上下文一致性
- Affected code: `scripts/` 启动脚本、`package.json`、`README.md`、`ENVIRONMENT_GUIDE.md`、必要时 `.env.example`

## ADDED Requirements
### Requirement: 统一开发态启动入口
系统 SHALL 提供一个由用户手动执行的单一开发态启动入口，用于统一最小门禁、数据库和缓存所需的关键运行上下文。

#### Scenario: 成功启动开发服务器
- **WHEN** 用户按文档执行统一的开发态启动入口
- **THEN** 启动流程会使用同一套认证、数据库和缓存运行上下文启动开发服务器
- **THEN** 浏览器访问受保护页面时，可用开发态有效凭据完成登录验证

#### Scenario: 缺少关键环境变量
- **WHEN** 用户执行开发态启动入口，但缺少关键认证或数据库环境变量
- **THEN** 启动流程应直接失败
- **THEN** 返回明确、可操作的缺失项提示，而不是让应用带着不完整上下文继续启动

### Requirement: 验证上下文一致性
系统 SHALL 确保浏览器验证与脚本验证对认证和数据库上下文的假设一致。

#### Scenario: 浏览器验证与脚本验证共用同一上下文
- **WHEN** 用户按统一入口启动开发服务器，并运行对应验证流程
- **THEN** 浏览器登录验证与脚本验证应基于同一套认证和数据库连接配置
- **THEN** 不再依赖临时运行时注入或一次性人工补参数

### Requirement: 保持最小门禁安全边界
系统 SHALL 在改进开发态易用性的同时保持现有最小门禁安全边界不变。

#### Scenario: 不绕过最小门禁
- **WHEN** 为开发态引入统一启动入口
- **THEN** 不得绕过 `middleware`
- **THEN** 不得取消登录门禁
- **THEN** 不得引入第二套认证路径、开发态后门或额外复杂认证框架

## MODIFIED Requirements
### Requirement: 本地开发启动方式
本地开发默认应通过显式、可重复执行的统一入口启动，而不是依赖裸 `npm run dev` 与隐式环境状态。
