# Phase04-03 观测与健康加固 Spec

## Why
当前仓库已经存在健康检查、错误日志和基础性能监测能力，但状态命名、信号出口和文档口径仍然分叉，导致运行问题排查依赖隐性知识。进入 `phase04-performance-and-ops-03-observability-and-health-hardening` 的目标，是在不引入重型平台的前提下，收口为一套最小、稳定、可解释的运行治理基线。

## What Changes
- 定义当前阶段唯一的最小可观察能力边界，明确健康检查、错误日志、基础性能指标各自负责回答什么问题
- 统一健康检查状态语义，避免 API 返回、内部检查器和文档各自维护第二套状态口径
- 明确错误日志主线与辅助能力边界，避免“内存日志”“文件日志”“控制台输出”并存但职责不清
- 明确基础性能指标主线与最小输出集合，确保问题定位入口可见、可解释
- 同步运行治理相关文档口径，使代码、脚本和文档对外说明一致

## Impact
- Affected specs: `phase04-performance-and-ops-*`、运行治理、健康检查、错误日志、基础性能指标
- Affected code: `src/app/api/health/route.ts`、`src/lib/health-checker.ts`、`src/lib/error-tracker.ts`、`src/lib/performance-monitor.ts`、必要时 `src/lib/api-error-handler.ts`、`README.md`、`DEPLOYMENT.md`

## ADDED Requirements
### Requirement: 统一最小可观察能力基线
系统 SHALL 提供一套最小、明确的运行治理基线，使开发者可以通过固定入口理解系统当前是否可用、最近是否有错误、关键请求是否出现明显性能退化。

#### Scenario: 通过固定入口定位运行状态
- **WHEN** 开发者需要判断当前实例是否健康
- **THEN** 系统提供固定且可解释的健康检查入口
- **AND** 返回结果中包含统一的整体状态语义
- **AND** 文档说明该入口与其他运行治理能力之间的关系

#### Scenario: 通过固定入口定位近期错误
- **WHEN** 开发者需要查看近期错误与失败请求
- **THEN** 系统提供明确的错误日志主线
- **AND** 文档说明哪些日志属于主要定位入口，哪些仅属于兼容或辅助能力

#### Scenario: 通过固定入口判断性能退化
- **WHEN** 开发者需要确认是否存在明显慢请求或错误率上升
- **THEN** 系统提供最小但稳定的基础性能指标
- **AND** 指标名称、来源和使用方式在代码与文档中保持一致

### Requirement: 统一健康状态语义
系统 SHALL 在健康检查 API、内部健康检查实现和相关文档中使用同一套整体状态语义，并明确子检查结果与整体状态的映射关系。

#### Scenario: 健康状态可跨实现对齐
- **WHEN** 健康检查返回数据库、内存、磁盘或其他子项结果
- **THEN** 开发者可以明确理解子项状态如何影响整体状态
- **AND** 不会出现“整体状态一套命名、子检查状态另一套命名、文档第三套命名”的分叉

### Requirement: 明确错误日志职责边界
系统 SHALL 明确错误日志的主要收集路径、保留方式和定位用途，并区分主要错误日志能力与临时控制台输出或兼容型日志能力。

#### Scenario: 错误日志主线清晰
- **WHEN** API 或运行时发生错误
- **THEN** 错误会进入可解释的主日志路径
- **AND** 文档说明该路径适合定位哪些问题
- **AND** 临时或兼容型日志能力不会被误读为当前主线

### Requirement: 明确基础性能指标主线
系统 SHALL 定义当前阶段的基础性能指标最小集合，并明确其采集范围、慢请求阈值和输出用途。

#### Scenario: 慢请求和错误率可被解释
- **WHEN** 开发者查看基础性能指标
- **THEN** 能看到至少覆盖请求量、响应时长、慢请求和错误率的最小指标集合
- **AND** 能理解这些指标来自哪条实现路径
- **AND** 不依赖隐性知识猜测其统计口径

## MODIFIED Requirements
### Requirement: Phase04 运行治理口径
`phase04-performance-and-ops-*` 的运行治理能力必须优先服务问题定位，并在 API、脚本、代码注释和文档说明之间保持同一套解释口径；不允许继续保留互相冲突的健康状态、日志主线和基础性能指标定义。

#### Scenario: 文档与实现一致
- **WHEN** 开发者阅读运行治理文档并查看对应代码
- **THEN** 能得到一致的状态命名、指标解释和定位入口
- **AND** 不需要通过历史上下文推断当前主线

## REMOVED Requirements
### Requirement: 隐性运行治理知识依赖
**Reason**: 当前阶段要求把问题定位入口显式化，不能继续依赖“知道某个内部类、某个页面或某个脚本的人”才能理解系统状态。
**Migration**: 将现有健康检查、错误日志和基础性能指标的说明统一回写到固定入口和相关文档中，并显式标注主线与兼容项边界。
