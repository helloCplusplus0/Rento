# project_skills.md

## 1. 合同锚点技能
- `Contract` 是租务事实主锚点，账单、续租、退租、抄表、房态变化都应优先围绕合同建模。
- 页面交互若与合同主链冲突，应优先修正表达，不轻易重写数据模型。
- 删除、编辑、续租、退租等动作必须首先判断对合同链和账务链的影响。

## 2. 多仪表建模技能
- 当前主线必须继续支持“一个房间可绑定多个仪表”。
- 不能为了轻量化而退回通用产品的“一房一电一冷一热”假设。
- `Meter` 作为独立资产存在，房间绑定关系服务于当前业务，不应吞掉历史。

## 3. 历史保留技能
- 对账单、抄表、合同、仪表历史，默认优先保留事实记录。
- 凡是会影响历史追溯的删除逻辑，都视为高风险改动。
- 若必须移除当前关联，应优先建模为解绑、停用、终止、归档，而不是物理删除。

## 4. 删除门禁技能
- 房间处于 `OCCUPIED` 或 `OVERDUE` 时，不应按普通空房直接删除。
- 合同未结束、账单未结清、仪表仍有关联历史时，删除操作必须被服务端门禁拦截。
- 删除门禁应由服务端规则保证，而不是只依赖前端按钮状态。

## 5. 账务语义技能
- 必须区分 `monthlyRent`、`totalRent`、押金、首期应收、账单应收额等不同语义。
- 支付周期（月付、季付、半年付、年付）应明确映射到账单生成策略，不允许 UI 有表达、后端无语义。
- 多仪表账单的聚合逻辑必须可解释、可追溯、可拆回明细。
- `pendingAmount`、`receivedAmount` 与 `BillStatus` 的组合语义必须稳定。

## 6. UI 承接技能
- 当前 `Rento` 前端 UI 展示效果已符合预期，原地重构时默认优先承接页面信息结构、组件表达、交互节奏与整体风格。
- 非必要不得擅自改变展示效果；允许的调整仅限宿主适配、明显 bug 修复、移动端可用性改善、最小信息架构优化，且每类调整都必须附最小技术适配说明或明确收益说明。
- 不允许借迁移之名重新设计整套视觉体系。

## 7. 原地重构承接技能
- 当前仓库中的旧实现代码就是后续重构的直接参考基线，不需要另行复制一份嵌入式源码区。
- 每个重构任务都必须写明参考来源文件或模块。
- 若某段旧实现只是在 `Next.js / Prisma / Redis / Docker-heavy` 前提下成立，应先拆出业务语义，再决定新的承载方式。
- 对原内嵌 `Rento-miniX/` 目录中的文档，已按要求完成有效规划内容吸收到根级真相源；后续若再出现同类目录，不得长期双线并存。

## 7.1 Hono 路线图冻结技能
- 当 `Hono` 版已经被明确为正式目标方案时，不应只冻结“首个正式实现阶段”，而应先冻结完整 Phase 路线图。
- 完整路线图至少应覆盖：应用壳与运行时基础、API 与认证骨架、领域服务迁移、数据访问层收口、部署主线切换。
- 在进入首个正式实现阶段 `/plan` 前，必须同步冻结模块级分类：哪些内容 `直接复用`、哪些需要 `包一层适配`、哪些 `必须重写`、哪些 `延后决策`。
- 对原 `Rento-miniX/` 目录中的历史规划文件，已给出文件级承接位；后续若再出现同类历史规划目录，也必须同步给出明确映射，避免只说“原则上已吸收”却没有具体承接位。

## 7.2 应用壳承接技能
- `phase07-app-shell-and-runtime-foundation` 的核心目标是先建立新前端应用壳与新运行时承接位，而不是一次性迁移全部业务页面和业务 API。
- 当前 `AppLayout`、导航壳、路由元数据、辅助页面治理口径、健康检查逻辑都应优先视为 `phase07` 的承接资产。
- `phase07` 允许通过最小技术适配把现有前端壳迁到 `Vite + React Router`，但不得借机重做 UI 或改写合同、账单、仪表、抄表主链语义。
- `phase07` 允许建立 `Hono + @hono/node-server` 的最小运行时入口与 `/api/health` 承接位，但完整认证骨架与正式业务 API 迁移应留给后续阶段。

## 8. 轻部署技能
- `Rento-miniX` 的核心目标之一是降低低配服务器部署门槛。
- 默认要求云端不构建，只运行预构建产物。
- 默认要求部署链低复杂度、低 I/O 峰值、低维护负担。
- 当推进到 `phase11` 时，默认冻结 `Caddy + systemd + Hono + PostgreSQL` 作为正式部署主线，避免重新回到 `Docker-heavy` 或多入口并存。

## 8.1 Prisma 保留技能
- 当前默认目标不是替换 `Prisma`，而是在保留 `Prisma + PostgreSQL` 的前提下完成旧 `Next.js` / legacy API / legacy PWA / Docker-heavy 运行线退出。
- 若后续再次评估 Prisma 替换，必须作为独立议题重新规划，不得与页面 parity、API 宿主切流或 legacy 退出绑定为同一阶段任务。

## 8.2 前端 parity 技能
- 当进入 `phase12 ~ phase16` 时，默认优先围绕页面 parity -> API parity -> PWA parity -> cutover 分层推进；其中 `phase12` 先冻结页面壳、页面装配、数据加载边界与正式路由承接位的迁移蓝图，而不是重做 UI 或重写业务语义。
- 新宿主优先复用现有 `src/components/pages/*`、`src/components/business/*`、`src/components/layout/*` 与 `src/components/ui/*`，只在技术适配确有必要时调整宿主绑定层。
- 旧 `src/app` 页面应优先作为页面原型与行为参考基线；迁移验收默认比较页面信息结构、导航节奏、表单交互、状态反馈与主链流程一致性。
- 任何页面迁移都必须以旧 `Rento` 源代码为直接原型；除显式批准的最小技术适配外，接近 `100%` 还原旧页面的信息结构、组件表达、导航节奏、表单交互、状态反馈与主链流程，是验收通过条件之一。
- 进入 `phase13-frontend-page-parity-implementation` 后，默认先完成 P0/P1 正式页面的真实 route module 承接、页面装配层落位、页面级加载/错态边界迁移与宿主绑定拆分，再把结果交给 `phase14` 承接 API/query parity。
- `phase13` 中出现 `next/navigation`、`next/link`、`generateMetadata()`、`notFound()`、`dynamic = 'force-dynamic'`、页面级 server query 与 Decimal 转换时，默认优先拆出宿主协议和 route-level 数据边界，而不是直接判定整页必须重写。
- `phase13` 的页面验收默认至少比较：首页快捷入口与导航节奏、列表页搜索/筛选与空态、详情页信息结构与错误恢复、编辑/新建页表单交互与提交后回跳、流程动作页状态反馈与历史语义保真。
- 若 `Rento-miniX` 当前页面仍包含迁移说明卡片、宿主标签、开发态状态卡、占位入口、重复快捷操作或与旧首页/旧列表页/旧详情页明显不一致的信息架构，应判定为“严重漂移”，必须回退验收结论并重新修复。

## 8.3 PWA 承接技能
- 当前 PWA 能力的价值已经在 `phase05` 冻结，后续目标不是取消 PWA，而是把安装、更新、最小离线兜底与发布口径迁到纯新主线。
- PWA 迁移默认保持最小受控策略：不因“追求离线能力”而缓存动态鉴权业务接口，不因“安装体验”而引入第二套前端宿主或第二套缓存真相源。

## 9. 迁移链治理技能
- 当前主线已是 PostgreSQL，但迁移链仍有 SQLite 时代遗留。
- `migration_lock.toml`、历史 SQL、兼容脚本需要被明确标记为“兼容项”而不是“当前最佳实践”。
- 在未设计迁移链整改方案前，不要贸然修改历史锁文件，以免破坏已有部署路径。

## 10. 重构验收技能
- “具备原地重构条件”至少包含三层：
- 工程层：`lint`、`type-check`、构建、健康检查路径可执行。
- 治理层：顶层文档完整、目录收口、真相源单一、内嵌规划目录不再争夺主线。
- 上线层：鉴权、安全门禁、环境卫生、迁移链一致性达到最低标准。

## 11. 查询与性能治理技能
- 关键列表与统计接口优先数据库侧过滤、分页、排序与聚合。
- 若存在全量查询后内存过滤，应优先评估是否能迁移到数据库侧，而不是先引入缓存。
- 性能优化应优先服务正式业务主链，不被演示页或基准页带偏。

## 12. 轻量观测治理技能
- 健康检查、错误日志与基础性能指标优先解决“能否定位问题”，而不是追求平台堆叠。
- 当前阶段允许补最小可观察能力，但不默认引入外部 APM、Tracing 或复杂告警编排。
- 运行治理口径必须在 API、脚本与文档之间保持一致。

## 13. 存量运行线治理技能
- 当前 `docker-compose.yml`、`nginx/nginx.conf`、`scripts/cloud-deploy.sh` 等仍服务于旧容器化运行线。
- 它们在 `phase06` 阶段应被视为参考输入和回滚基线，而不是未来 `Rento-miniX` 正式部署主线。
- 对外说明必须区分“当前存量可运行”与“未来轻量主线目标”，避免混写。
- 当进入 `phase11` 时，默认要求把这些资产显式降级为 legacy 回滚基线，并写清保留条件、退出条件与不得继续扩写的边界。
