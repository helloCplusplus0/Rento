# global_skills.md

> 职责：跨项目、跨阶段通用方法论。
> 本文件只承接“怎么做”，不承接 Rento 专属业务规则与阶段状态。

## 1. 需求澄清技能
- 进入任务前先明确目标、边界、约束、成功标准与非目标。
- 对模糊诉求先拆成可验证任务，再进入设计或实现。
- 优先区分“当前必须做的动作”和“未来可扩展设想”。

## 2. 单一真相源技能
- 坚持单仓库、单主线、单一真相源，避免第二套默认入口、第二套规划目录或第二套正式部署说明。
- 每份根级文档只回答一类问题，不让多个文档重复承载同一结论。
- 发生结构、部署、安全或状态变化时，只在对应真相源更新，其他文档改为引用而不是复述。

## 3. 规划技能
- 当推进方向不明确或范围较大时，优先进入 `/plan`。
- 先在 `plan.md` 做整体规划，再对当前执行 `phase` 进入 `/plan`。
- 当目标较小、边界集中时，可只规划一个 `phase`；当目标明确但跨度大时，可一次规划多个连续 `phase`。
- `/plan` 前先同步根级真相源，再产出阶段级 `architecture_plan`、`dev_plan`，必要时补 `shared_baseline`。
- 阶段级文档产出后先停下等待审核，未经批准不直接进入 `/spec` 或实现。

## 4. 实施技能
- 进入实施后，按已批准 `dev_plan` 的顺序推进，不顺手扩大范围。
- 优先最小边界、分阶段推进，避免把 UI、框架、数据层、部署一起大爆炸改写。
- 历史实现若仍有参考价值，先提炼语义与边界，再决定复用、适配、重写或延后。
- `phase` 场景默认按“`plan.md` 规划 -> 当前 `phase` 执行 `/plan` -> 产出三件套 -> 审核 -> 按 `dev_plan` 逐子任务 `/spec` -> 实施 -> 验收”推进。
- 非 `phase` 场景默认按 fix 工作流推进：先记录 issue，再产出 analysis，再进入 fix 的 `/spec`、实施与验收。

## 5. 文档治理技能
- `AGENTS.md` 负责入口摘要与协作约束。
- `plan.md` 负责阶段路线图与状态。
- `project_rules.md` 负责硬规则与门禁。
- `architecture_map.md` 负责目录、模块与运行时落点。
- `global_skills.md` 负责通用方法；`project_skills.md` 负责项目专属实践。
- 阶段级设计文档默认使用 `docs/phaseX_<workflow>_architecture_plan.md`、`docs/phaseX_<workflow>_dev_plan.md` 和可选 `shared_baseline`。

## 6. 校验技能
- 文档变更至少检查互链、被引用路径与根级真相源一致性。
- 实现变更至少检查 `lint`、`type-check`、构建、健康检查与核心 smoke test。
- 对部署与运行相关改动，优先验证是否可运行、可维护、可回退，而不是追求复杂基础设施。

## 7. Fix 工作流技能
- 当前没有新的 `phase` 规划时，默认进入 `fix` 工作模式。
- fix 工作流先填写 `docs/fix/fix_*_issue_*.md`，记录现象、复现条件、影响面与初步观察。
- 再基于 issue 文档产出 `docs/fix/fix_*_analysis_*.md`，完成根因、证据链、候选方案、推荐方案、验收标准与回滚条件。
- 若某个 fix 已明显演化为结构性问题或长期迁移议题，应停止继续按普通 fix 推进，并回到 `plan.md` 评估是否新开 `phase`。

## 8. 发布与回退治理技能
- 任何影响公网暴露面的变更，都必须先定义门禁与回退条件。
- 对兼容逻辑、桥接逻辑与新旧并存逻辑，必须写清存在原因、当前职责与退出条件。
- 发布前优先确认环境变量、健康检查、部署脚本与核心 smoke test 已被冻结为单一解释。

## 9. 问题分级技能
- `fix-now`：直接破坏当前真相源或阻断主线推进的问题。
- `defer-post-phase`：不阻断当前闭环，但必须进入后续计划的问题。
- `escalate-architecture`：已超出局部修补，需要单独设计或新阶段处理的问题。
