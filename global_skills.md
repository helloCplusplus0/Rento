# global_skills.md

## 1. 需求澄清技能
- 在进入任一阶段前，先明确目标、边界、约束、成功标准与非目标。
- 优先区分“当前阶段必须做的切换动作”与“未来可扩展设想”，避免把 `Rento-miniX` 再次做成高复杂度项目。
- 对模糊诉求先拆成可验证任务，再进入设计或编码。

## 2. 原地重构工作流技能
- 当前仓库采用：

```text
reference-driven in-place replatform workflow
参考基线驱动的原地重构工作流
```

- 旧 `Rento` 源码不是被抛弃的历史材料，而是当前仓库中后续重构的直接参考基线。
- 默认先冻结“参考来源、复用内容、调整内容、舍弃内容及其原因”，再进入实现。
- 若旧实现与新主线目标冲突，优先在阶段文档中写清冲突与取舍，不在实现阶段边做边改。

## 3. 边界控制技能
- 坚持低复杂度、单仓库、单主线、单一真相源。
- 对超出当前阶段的问题，优先外置到后续 `phase`，而不是在当前阶段顺手扩写。
- 避免把 UI 承接、后端骨架迁移、ORM 切换、部署重构一次性绑定为单一大爆炸任务。

## 4. 文档治理技能
- `AGENTS.md` 负责入口摘要与执行总约束。
- `project_rules.md` 负责刚性规则、发布门禁与禁止事项。
- `plan.md` 负责整体项目推进预览，不承接细粒度子任务。
- `architecture_map.md` 负责当前仓库结构、运行入口与重构承接位说明。
- `global_skills.md` 负责通用方法论，`project_skills.md` 负责项目专属实践。
- 阶段级设计文档默认拆为 `docs/phaseX_<workflow>_architecture_plan.md` 与 `docs/phaseX_<workflow>_dev_plan.md`；存在共享边界时再补 `shared_baseline`。
- 当运行入口、部署方式、数据库主线、安全边界或目录结构变化时，必须同步相关文档。

## 5. 阶段规划技能
- 当推进方向不明确时，优先进入 `/plan`，基于当前源代码、顶层真相源和现有阶段结论生成阶段推进文档。
- `/plan` 的最低交付包括：先同步顶层规范文档，再产出该阶段的 `architecture_plan`、`dev_plan` 与 `shared_baseline`。
- 阶段级文档产出后必须停止工作流，等待用户审核；未经明确批准，不得直接进入实现、`/spec` 或扩大任务范围。
- 进入实施阶段后，必须严格按 `dev_plan` 的子任务顺序逐个进入 `/spec`、开发、验收、提交与推送。
- 每个已批准 `spec` 子任务在开发完成后，必须额外指定独立子代理执行审核验收；只有在子代理审核结论明确通过后，才允许把该子任务标记完成，并继续提交与推送。
- 对 `phase06-minix-replatform-01-*`，必须先冻结真相源切换、UI 承接、参考基线与重构顺序，再进入任何实现子任务。
- 对“目标技术栈已明确、且旧实现为直接参考基线”的原地重构项目，应优先冻结完整 Phase 路线图、模块级 `直接复用 / 包一层适配 / 必须重写 / 延后决策` 分类，以及历史规划材料的文件级吸收映射，再进入首个正式实现阶段 `/plan`。
- 当推进到 `phase07-app-shell-and-runtime-foundation` 时，必须先冻结前端路由方案、开发拓扑、并行壳切入策略、实现目录与最小环境变量口径，再进入该阶段 `/spec`。
- 当推进到 `phase12-frontend-parity-and-shell-cutover` 时，必须先冻结旧页面到 `src/minix` 的映射表、页面装配复用策略、UI 保真边界、`Prisma + PostgreSQL` 保留口径，以及 `phase12 ~ phase16` 的完整路线图，再进入该阶段 `/spec`。
- 当推进到 `phase13-frontend-page-parity-implementation` 时，必须先冻结 P0/P1 页面切片顺序、route module 组织方式、页面装配/数据加载边界、宿主绑定拆分策略、页面级加载/错态边界与浏览器验收基线，再进入该阶段 `/spec`。

## 6. 迁移保真技能
- 涉及合同、账单、支付周期、仪表、抄表主链时，优先验证业务真实、状态可解释、历史可追溯，而不是先追求形式上的极简。
- 对每次迁移至少检查：页面预展示链路、服务端执行链路、数据模型映射、历史数据影响、其他复用路径。
- 若发现某段历史实现与正式业务语义冲突，应先写清楚“保留、调整还是舍弃”的理由。
- 当前 `Rento` UI 展示效果已符合预期，默认冻结；若需调整，必须先说明收益与范围。
- 当进入 `phase13` 的真实页面迁移实施阶段时，优先把旧 `src/app/**/page.tsx` 中的宿主绑定、服务端 query 入口、`generateMetadata()`、`notFound()` 与 `next/*` 路由协议拆离到新宿主 route module，而不是把页面主体表达层整目录搬运到 `src/minix`。

## 7. 工程校验技能
- 对阶段切换至少检查：文档是否一致、仓库远端是否正确、技术栈是否仍匹配目标方案、验证路径是否可执行。
- 对实现阶段至少检查：`lint`、`type-check`、构建、健康检查与核心 smoke test。
- 对部署与运行阶段，优先验证低配服务器是否可运行、可维护、可回滚，而不是追求花哨基础设施。

## 8. 发布与回滚治理技能
- 任何影响公网暴露面的变更，都必须先定义门禁与回滚条件。
- 对历史兼容逻辑、迁移桥接逻辑与新旧并存逻辑，必须写明存在原因、当前作用与退出条件。
- 发布前优先确认环境变量、健康检查、部署脚本与核心业务 smoke test 已冻结。
- 当推进到 `phase11-deployment-cutover-and-cutline-closure` 时，优先冻结正式部署主线、legacy 回滚基线、环境模板、健康检查与发布门禁，再进入任何部署实现。

## 9. 文档-实现一致性扫描技能
- 在阶段切换前，检查顶层文档是否准确描述当前代码、目录与仓库状态。
- 对历史遗留要明确标注“仍存在但已降级为兼容项/参考项”，避免被误读成当前主线。
- 对归档文档维持只读边界，不让其重新争夺当前真相源地位。

## 10. 问题分级技能
- 对发现的问题优先按 `fix-now / defer-post-phase / escalate-architecture` 分级。
- `fix-now`：会阻断当前仓库主线切换或直接破坏真相源的问题。
- `defer-post-phase`：不影响当前阶段闭环，但后续必须进入计划的问题。
- `escalate-architecture`：已经超出局部修补，需要单独 phase 收口的问题。
- 对 `phase11` 相关问题，可额外优先识别两类高风险项：
- `cutover-blocker`：会导致正式部署主线无法替代 legacy 运行线的问题，例如服务端产物链缺失、环境模板失真、健康检查失效。
- `rollback-gap`：会导致切线后无法回退到已知可用基线的问题，例如 legacy 资产职责不清、回滚路径与发布门禁混写。
- 对 `phase12 ~ phase16` 相关问题，可额外优先识别两类高风险项：
- `parity-blocker`：会导致纯新主线无法完整承接旧页面、旧 API 或旧 PWA 能力的问题，例如页面映射缺失、保真边界不清或 retained-legacy 无法退出。
- `ui-drift`：会导致迁移过程脱离旧 `Rento` 页面原型、重新设计视觉体系或破坏现有交互节奏的问题。
