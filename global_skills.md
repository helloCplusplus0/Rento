# global_skills.md

## 1. 需求澄清技能
- 在开始设计前先确认目标、边界、约束和成功标准。
- 优先区分“当前重启所需”与“未来可扩展设想”，避免把讨论直接升级成复杂实现。
- 对模糊诉求先拆成可验证任务，再进入设计或编码。

## 2. 边界控制技能
- 坚持低复杂度、单仓库、单主链、单一真相源。
- 对超出当前阶段的问题，优先外置为后续 phase，而不是在当前任务内顺手扩写。
- 发现历史方案与当前主线冲突时，优先清理双重真相，不长期并存。

## 3. 文档治理技能
- `AGENTS.md` 负责入口摘要与执行总约束。
- `project_rules.md` 负责刚性规则、发布门禁与禁止事项。
- `plan.md` 负责默认阶段顺序、目标与验收结论。
- `architecture_map.md` 负责代码结构、运行入口与目录说明。
- `global_skills.md` 负责通用方法论，`project_skills.md` 负责项目专属实践。
- 阶段级设计文档默认拆为 `docs/phaseX_<workflow>_architecture_plan.md` 与 `docs/phaseX_<workflow>_dev_plan.md`；存在共享边界时再补 `shared_baseline`。
- `spec` 目录默认使用 `.trae/specs/phaseX-<workflow>-<nn>-<task-name>/` 模板，避免命名与阶段语义漂移。
- 当运行入口、数据库主线、安全边界或目录结构变化时，必须同步相关文档。

## 4. 低复杂度架构技能
- 优先直接、清晰、可回滚的方案，不为“也许会需要”提前引入复杂组件。
- 对需要后续演进的问题，用 phase 切分，而不是在一个任务内混合治理、重构和新功能。
- 在无法同时满足“快”和“稳”时，优先保持业务主链稳定。
- 在确保实现符合最佳实践的前提下，始终优先保持轻量、高效、灵活、可扩展、低复杂度、低耦合的风格。
- 对任何明显提高耦合度、心智负担或长期维护成本的方案，优先降级为后续 phase，而不是并入当前任务。

## 5. 阶段规划技能
- 当用户对推进方向不明确时，优先进入 `/plan`，基于当前源代码、顶层真相源和现有阶段结论，在 `.trae/documents/` 下生成阶段推进文档。
- `/plan` 的最低交付包括：先同步顶层规范文档，再产出该阶段的 `architecture_plan` 与 `dev_plan`；必要时补充 `shared_baseline`。
- 阶段级文档产出后必须停止工作流，等待用户审核；未经明确批准，不得直接进入实现、`/spec` 或扩大任务范围。
- 进入实施阶段后，必须严格按 `dev_plan` 的子任务顺序逐个进入 `/spec`、开发、验收、提交与推送。

## 6. 工程校验技能
- 对重启准备至少检查：`lint`、`type-check`、`build`、`prisma validate`、环境模板与真实实现一致性。
- 对命令输出不只看 exit code，也要注意警告和重复噪音，例如过时命令、重复注册、网络重试。
- 对“能编译”与“可重启”分开判断，前者只是最低条件，后者还要看治理和发布门禁。

## 7. 发布与回滚治理技能
- 任何会影响公网暴露面的变更，必须先定义门禁和回滚条件。
- 对历史兼容逻辑要写明存在原因、当前作用和退出条件。
- 发布前优先确认环境变量、迁移链、健康检查和核心业务 smoke test。

## 8. 问题分级技能
- 对发现的问题优先按 `fix-now / defer-post-phase / escalate-architecture` 分级。
- `fix-now`：会阻断重启或直接破坏当前真相源的问题。
- `defer-post-phase`：不影响当前阶段闭环，但后续必须进入计划的问题。
- `escalate-architecture`：已经超出局部修补，需要单独 phase 收口的问题。

## 9. 文档-实现一致性扫描技能
- 在阶段切换前，检查顶层文档是否仍准确描述当前代码与目录。
- 对历史遗留要明确标注“仍存在但已降级为兼容项”，避免被误读成当前主线。
- 对归档文档维持只读边界，不让其重新争夺当前真相源地位。
